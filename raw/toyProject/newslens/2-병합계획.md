## 0. 한눈에 — 결론부터

- **베이스는 팀원 기획 100% 그대로.** 스택·기능·8일 일정 유지.
- 스택은 **Spring Boot로 통일**. (내 Node+Vue 계획은 접고, 학습 목표만 Spring으로 이전)

## 2. 겹친 것 (병합 접점)

두 기획이 **같은 기술을 다른 각도로** 만나는 지점이 정확히 병합 포인트다.

| 기술 | ① 팀원 기획에서의 쓰임 | ② 내 계획에서의 쓰임 | 병합 판단 |
|---|---|---|---|
| **Redis** | 캐싱 (피드/카테고리 TTL) | Pub/Sub 다중서버 확장 | 8일엔 **캐싱만**. Pub/Sub는 시즌2 |
| **실시간** | SSE (단방향 알림) | WebSocket (양방향) | **SSE 유지**. WebSocket 미채택 |
| **인증** | ❌ 기획에 없음 | 소켓 JWT 인증 | **스트레치** — 8일엔 자리만 예약, 구현은 여유 시/시즌2 (6·7장) |
| **배포/인프라** | Docker Compose → Railway | Docker→AWS→k8s 사다리 | 8일엔 Compose+Railway. AWS/k8s는 시즌2 |
| **동시성/상태관리** | (약함) | SseEmitter·접속자 관리 | 내 학습 관점을 SSE 운영에 부분 적용 |

→ **8일 코어에 실제로 병합되는 것은 "스택 통일 + JWT 자리 예약"** 정도다. JWT 본구현·나머지 겹침은 "8일 본체엔 빼고 여유 시/시즌2로 미룬다"는 정리.

## 3. 다른 것 (8일에서 제외 → 시즌2 보관)

내 계획에만 있고 팀원 기획엔 없으며, **이번 8일에는 넣지 않는** 것들. 버리는 게 아니라 **같은 코드베이스 위에 개인 확장으로 남긴다.**

- **WebSocket 양방향 통신** — SSE로 충분한 단방향 알림이라 8일엔 불필요. 추후 양방향 기능(예: 실시간 댓글) 생기면 추가 검토.
- **Redis Pub/Sub 다중서버 확장** — SSE를 서버 2대로 늘리면 한쪽 서버 구독자에게 다른 서버 이벤트가 안 가서 브로드캐스트가 깨진다. 이 "깨짐"을 직접 체험한 뒤 Redis Pub/Sub로 푸는 게 시즌2의 핵심 학습.
- **인프라 사다리 (AWS → minikube → EKS)** — 8일엔 Docker Compose + Railway까지만. 그 위에 AWS 배포 → 로컬 k8s → EKS로 올리는 게 시즌2.

> 시즌2는 **개인 포트폴리오**로, 8일 팀 결과물이 나온 뒤 이찬호가 단독으로 얹는다. 지금은 설계만 비워둔다.

## 4. 병합 확정 사항 (3가지)

1. **스택 통일** — Spring Boot 3.x 채택. 내 Node+Socket.IO 계획은 접고, "WebSocket/인증/동시성을 직접 구현해본다"는 **학습 목표만 Spring 위로 이전**. (내 plan.md에 이미 "Spring 대안 가능"으로 명시돼 있어 충돌 없음.)
2. **실시간 = SSE** — 새 기사 알림은 단방향이라 SSE가 적합. WebSocket 미채택.
3. **JWT 인증 = 스트레치 목표** — 8일 코어는 공개 API. 설계 단계에서 **User 테이블 자리 + 보호 경로 구조만 예약**하고, 구현은 여유 시(Day7~8) 또는 시즌2. "넣었다 빼는" 낭비 없이 양쪽을 열어두는 선택.

## 6. JWT를 붙였을 때 얻는 것 (왜 자리를 남기나)

팀원 기획엔 인증이 없다. 인증 없는 API는 데모로는 돌지만 "실서비스" 느낌이 약하다. JWT를 구현하면(스트레치/시즌2) 얻는 것:

1. **포트폴리오 완성도** — 회원/로그인 → 토큰 발급 → 보호된 API 흐름이 있으면 "실제 서비스를 만들어봤다"가 성립. 인증 없는 공개 API와 체감이 크게 다르다.
2. **SSE 인증의 특수성 학습 (가장 어필되는 지점)** — 브라우저 `EventSource`는 커스텀 헤더(`Authorization`)를 실을 수 없다. 그래서 토큰을 **쿼리파라미터 또는 쿠키**로 넘겨야 하고, 연결(핸드셰이크) 시점에 검증해야 한다. 이건 내 원래 계획의 "소켓 JWT 인증 함정"이 **SSE로 치환된 것** — 일반 HTTP 인증과 타이밍·방식이 다른 경험을 그대로 얻는다. (면접에서 말할 거리)
3. **Spring Security 필터체인 실무 경험** — 토큰 파싱·검증을 필터로 끼우는 표준 패턴을 직접 구성.
4. **Gemini 무료 쿼터 보호의 근거** — RAG/검색은 Gemini를 호출(하루 1,500 무료). 인증으로 사용자를 식별하면 **사용자별 호출 제한(rate limit)**을 걸 명분이 생긴다. (구현 시점에 식별→제한 순으로.)
5. **시즌2 개인화의 토대** — "내가 구독한 카테고리만 알림", "내 검색 기록" 같은 확장이 사용자 식별 위에서만 가능. 지금 인증을 깔아두면 나중이 열린다.

> 즉 JWT는 **고효과(완성도 + 학습 + 확장성)** 지만 8일 코어의 RAG 시간을 잠식할 수 있어 **스트레치로 두고 설계 자리만 예약**한다. 위 가치들은 구현하는 순간(여유 시/시즌2) 그대로 얻는다.

## 7. JWT 적용 지점 (구체 — 스트레치/시즌2 구현 시)

> 아래는 JWT를 **실제 구현할 때(여유 시 Day7~8 또는 시즌2)** 손대는 곳이다. 8일 코어에선 Day1~2에 ①의 `users` 테이블 자리와 보호 경로 구조만 예약하고, 실제 검증 로직은 넣지 않는다.

손대는 곳은 정확히 세 군데다.

| 대상 | 추가 내용 | 난이도/메모 |
|---|---|---|
| **User 도메인 + 로그인** | `users` 테이블, `signup`/`login` → JWT(Access Token) 발급. 최소 구성(리프레시 토큰은 시즌2). | 낮음. 표준 패턴 |
| **REST API 보호** | `SecurityConfig` + JWT 검증 필터. `/api/feed`, `/api/search`, `/api/articles/**` 등에 `Authorization: Bearer` 검증. | 낮음. 한 번 만들면 전역 적용 |
| **SSE 구독 인증** | `/api/subscribe` 연결 시 토큰 검증. `EventSource`가 헤더를 못 실으므로 **쿼리파라미터(`?token=`) 또는 쿠키**로 전달받아 검증. | **중간 — 학습 포인트.** 헤더 못 쓰는 타이밍 이슈 |

**공개로 둘 것 (인증 제외)**: `signup`, `login`, Swagger UI, (선택) 헬스체크. 나머지는 보호.

## 8. 업무 분담 제안 (재조정안)

> 기획서의 Developer A/B는 임의 구분이었으므로, **일감 무게와 의존 관계**를 기준으로 다시 나눈다.
> 핵심 원칙: **(1) 데이터 흐름 방향으로 가른다**(쓰기 라인 vs 읽기 라인) → 코드 충돌 최소화.
> **(2) 같이 깔아야 할 기반은 초반에 둘이 함께** 만든다.

### 8-1. 공통 기반 — Day 1~2에 **둘이 함께** (페어로 권장)

여기서 갈라지기 전에 같이 합의·구축해야 충돌이 안 난다.

- **DB 스키마 합의** — `articles`(원문+요약+태그+벡터), `users`, 카테고리 구조
- **Docker Compose 골격** — Postgres+pgvector, Redis 컨테이너 (앱은 각자 로컬 실행으로 시작)
- **Gemini API 클라이언트 래퍼** — 둘 다 쓰는 공용 모듈 (수집=처리용, 검색=RAG용). **한 명이 만들고 공유**
- **공통 응답/예외 포맷, 패키지 구조, Git 브랜치 전략**
- **(스트레치 대비) JWT 자리 예약** — `users` 테이블 스키마와 보호 경로 구조만 설계에 비워둔다. 구현은 안 함.

### 8-2. 분담 — Day 3 이후

| | **개발자 1 — 수집·처리·인프라 라인 (쓰기)** | **개발자 2 — 조회·검색·실시간 라인 (읽기, +인증 스트레치)** |
|---|---|---|
| 핵심 | 데이터를 **만들어 넣는** 쪽 | 데이터를 **꺼내 보여주는** 쪽 |
| 담당 | • RSS/News 수집 배치 + 스케줄러<br>• URL 해시 중복 제거<br>• Gemini LLM 처리 (요약·태깅·임베딩)<br>• DB 저장 + 수집완료 **이벤트 발행**<br>• Docker Compose 마무리<br>• 배포 (Railway/Render) | • Feed/Articles/Categories API<br>• Redis 캐싱 레이어<br>• pgvector **RAG 검색** API<br>• **SSE 알림** (이벤트 구독→푸시)<br>• Swagger 문서화<br>• *(스트레치) JWT+User 인증 모듈·SecurityConfig·SSE 토큰검증* |
| 데모 UI | (Day 7~8) 개발자 1이 React 데모 보조 | (Day 7~8) 개발자 2 API 연결 주도 |

### 8-3. 왜 이렇게 나눴나 (근거)

- **충돌 최소화**: 쓰기 라인(배치→DB)과 읽기 라인(DB→API)은 **DB 스키마만 합의되면 거의 안 부딪힌다.** 같은 컨트롤러를 둘이 만지는 일이 없게.
- **JWT(스트레치)는 개발자 2 영역**: 구현하게 되면 인증 필터가 API 앞단에 붙고 SSE 토큰 검증도 개발자 2의 SSE 영역이라 한 사람이 들고 가는 게 깔끔. 8일 코어에선 구현하지 않고, 여유 시/시즌2에 개발자 2가 담당.
- **무게 균형**: 개발자 1의 **Gemini LLM 통합**(요약·태깅·임베딩, 가장 무거운 단일 작업)과 개발자 2의 **인증+RAG+SSE**가 대략 비슷한 무게. 균형 맞추려고 **Docker/배포/데모UI 보조를 개발자 1쪽**으로 보냄.
- **의존 관계 1개만 주의**: 개발자 2의 RAG가 개발자 1이 만든 **Gemini 클라이언트 래퍼**에 의존 → 그래서 8-1에서 **공용 래퍼를 먼저** 만든다.

### 8-4. 이찬호의 역할

- 위 두 라인 중 어느 쪽을 맡든 무방하나, **시즌2의 JWT·실시간 확장을 직접 가져가려면 개발자 2** 가 자연스럽다 (내 학습 목표인 "인증/동시성/실시간"이 모두 읽기 라인에 모여 있음).
- 최종 배정은 팀원과 협의 — 이 문서는 **두 라인의 내용만 확정**하고, 누가 어느 라인인지는 협의 표시.

## 11. 이찬호 제안 사항 (팀 협의용 — 베이스 변경 아님)

> 아래는 **확정이 아니라 팀 미팅에서 제안해볼 후보**다. 팀원 베이스 기획을 바꾸지 않으며, 합의되면 그때 반영한다.

### 11-1. Spring AI 도입 제안 ⭐ (검증 완료)

Gemini를 직접 HTTP 호출 + pgvector를 SQL 직접 다루는 대신, **Spring 공식 `spring-ai-starter-model-google-genai`** 로 RAG 배선을 줄이자는 제안.

- **무료 티어 연동 검증 완료** — AI Studio API 키(빌링 불필요)로 동작. 단 `project-id`/`location`을 넣으면 Vertex 유료 모드로 전환되니 **절대 넣지 않는다.**
- 얻는 것: RAG 배선 코드 대폭 감소(8일에 유리) + "Spring AI로 RAG 구현"이라는 AX 포트폴리오 신호.
- 백업 경로: 막히면 Gemini의 OpenAI 호환 엔드포인트 + `spring-ai-starter-model-openai`.

```properties
# 채팅(요약·태깅·RAG 답변) + 임베딩(벡터) — 모두 무료 티어
spring.ai.google.genai.api-key=${GEMINI_API_KEY}
spring.ai.google.genai.chat.model=gemini-2.0-flash
spring.ai.google.genai.embedding.api-key=${GEMINI_API_KEY}
spring.ai.google.genai.embedding.text.model=text-embedding-004
# project-id / location 은 넣지 않는다 (넣으면 Vertex 유료 모드)
```

| 항목 | 확정 후보값 | 비고 |
|---|---|---|
| 임베딩 모델 | `text-embedding-004` | 기획에 비어있던 칸 |
| **벡터 차원** | **768** | → pgvector 컬럼 `vector(768)`로 생성 (미리 못 박아야 안 깨짐) |
| 채팅 모델 | `gemini-2.0-flash` | 무료 1,500/day |
| RAG 최적화 | `task-type` 자동 (`RETRIEVAL_DOCUMENT`/`RETRIEVAL_QUERY`) | Spring AI가 색인/검색 구분 |

### 11-4. 도메인 · 수집 소스 · LLM 처리 방식 제안

| 항목 | 제안 | 근거 |
|---|---|---|
| **도메인** | **AI·테크 동향** | 타깃이 테크·AX → "AI로 AI 뉴스를 큐레이션" 서사 자기강화. AI 뉴스는 빠르게 쏟아져 큐레이션 가치가 드러남. 니치라 카테고리 설계도 단순 |
| **수집 소스** | **The Guardian API** (영문, 전문 제공) | 외신이 국내보다 소식이 빠름. 본문 *전문* 제공이라 요약·임베딩 품질 ↑. 무료·배포 안전·API 깔끔 |
| **번역/처리** | "번역+요약+태깅"을 **1콜로 합침** (structured output) + 임베딩 1콜 = 기사당 **2요청** | 전문 통째 번역 X — 영문을 한국어로 *요약*하면 그게 곧 번역+요약. 원문은 영어 보관+출처 링크 |
| **RAG 검색** | 한국어 요약을 임베딩 → 한국어 질문/답변 | 한-한 매칭 직관적. (대안: 영문 본문 cross-lingual 임베딩) |

**무료 티어 안전성** (gemini-2.0-flash 기준)
- 병목은 토큰이 아니라 **요청 수**: RPD ~1,500 / RPM 5~15 / TPM 1M(사실상 안 닿음). 임베딩은 별도 쿼터.
- 견적: 하루 신규 ~100건 처리 = 채팅 100 + 임베딩 100 → **RPD의 7% 수준**. 하루 700건까지 무료로 가능.
- 주의점은 **RPM(분당 5~15)** → 배치 시 한 번에 쏟지 말고 **큐로 throttle/backoff**. (이 자체가 면접 어필 포인트)

> ❌ 피할 안티패턴: ① 번역/요약/태깅/임베딩을 4번 따로 호출 ② 전문 통째 번역(출력 토큰 낭비)
> ⚠️ NewsAPI.org는 무료 플랜이 **localhost 전용**이라 배포 시 막힘 — 후보 제외.

> ❗ LLM은 **Gemini Flash 단독**으로 충분(이 견적 기준). 기획서의 "Groq 병행" 질문은 불필요로 정리.
