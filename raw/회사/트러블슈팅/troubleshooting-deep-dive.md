> **💡 본 문서는 면접 시 기술 딥다이브 질문에 대비한 예상 스크립트이자, 향후 웹 포트폴리오의 상세 모달용으로 작성된 가상 시나리오입니다.**
> 이찬호 님의 실제 업무 경험 및 디테일에 따라 데이터 수치나 설계 구조를 필요에 맞춰 피드백해 주시면 즉시 반영하겠습니다.

## 1️⃣ [데이터 리니지] 계층형 조회 성능 저하 및 무한 루프(Cycle) 해결

### 1. Trouble (문제 상황)
*   사용자들의 동적인 메뉴 데이터 입력 과정에서 데이터의 A ➡️ B ➡️ C ➡️ A 와 같은 **순환 참조(Cycle)** 관계가 간헐적으로 발생했습니다.
*   이 상태에서 데이터 리니지 양방향 계보를 파악하기 위해 Oracle 계층형 쿼리(`START WITH... CONNECT BY`)를 실행할 때, 데이터베이스 서버가 순환 루프에 빠져 **`ORA-01436: CONNECT BY loop in user data`** 오류를 뿜으며 데이터베이스 CPU 커넥션을 고갈시키는 치명적인 장애가 발생했습니다.
*   또한, 100만 건 이상의 근로자 안전 데이터 테이블에 계층형 쿼리를 날릴 시, 깊이(Depth)가 깊어질수록 풀 테이블 스캔(Full Table Scan) 횟수가 누적되어 응답 시간이 8초 이상 지연되는 문제가 동반되었습니다.

### 2. Deep-Dive & Trade-off (원인 분석 및 대안 검토)
*   **대안 A: 애플리케이션 레이어에서 DFS/BFS 알고리즘으로 루프 검증**
    *   *장점*: DB 부하를 차단하고 비즈니스 로직 단에서 깔끔하게 제어 가능.
    *   *단점*: 대용량 데이터를 메모리에 모두 올려 노드를 탐색해야 하므로 WAS 메모리 압박이 심해지고 API 네트워크 오버헤드가 큼.
*   **대안 B: DB 스키마 단 유효성 제약 설정 및 Oracle `NOCYCLE` 키워드 활용 (채택)**
    *   *이유*: 관계형 데이터베이스 엔진 자체에서 제공하는 사이클 감지 기능을 최적화하여 구현하는 것이 메모리 효율 및 속도 면에서 압도적이라고 판단.

### 3. Resolution (해결 방안)
*   **순환 루프 방어**: 계층형 쿼리 작성 시 `CONNECT BY NOCYCLE` 구문을 적용하여 순환이 발견된 노드는 더 이상 전개하지 않고 계보 탐색을 즉시 중단하도록 예외 처리.
*   **계보 상태 필드(Is_Cycle) 가시화**: `CONNECT_BY_ISCYCLE` 가상 컬럼을 활용하여 어떤 워크플로우 단계에서 무한 루프가 발생하여 규제 증적 경로가 끊겼는지를 사용자 단에 경고 메시지로 노출.
*   **인덱스 튜닝**: `PARENT_ID`와 `ENTITY_TYPE`, `ORIGIN_ID` 컬럼에 복합 인덱스(Composite Index)를 구성하고, 역방향 조회 쿼리가 인덱스 Range Scan을 탈 수 있도록 실행 계획(Execution Plan) 분석 및 최적화 진행.

### 4. Result (결과)
*   무한 루프로 인한 커넥션 고갈 장애율 **0%**로 해결.
*   데이터 조회 쿼리 성능 향상: 평균 8.4초 ➡️ **0.15초 이내**로 98% 단축.

## 2️⃣ [멀티테넌시] Dynamic Routing 시 ThreadLocal 메모리 누수 방지

### 2. Trouble (문제 상황)
*   한화오션 등의 대기업 요구사항을 분리 격리하기 위해 인터셉터(Interceptor)와 AOP를 도입하여, 각 요청 헤더의 `X-Tenant-ID`를 파싱해 런타임에 동적으로 데이터 소스(RoutingDataSource)를 선택하도록 처리했습니다.
*   이때 테넌트 정보를 WAS의 `ThreadLocal` 콘텍스트에 저장하여 꺼내 쓰도록 설계했는데, 트래픽이 집중되는 피크 타임에 특정 테넌트의 정보가 다른 테넌트의 화면에 노출되는 **크로스 테넌시(Cross-Tenancy) 데이터 오염 및 유출 사고**가 불규칙하게 발생했습니다.
*   동시에 톰캣(Tomcat) 서버 모니터링 중 Old 영역의 메모리 점유율이 복구되지 않고 서서히 증가하는 메모리 누수(Memory Leak) 현상이 진단되었습니다.

### 2. Deep-Dive & Trade-off (원인 분석 및 대안 검토)
*   **원인**: WAS는 쓰레드 풀(Thread Pool) 모델을 사용하므로 한 번 사용된 쓰레드가 소멸하지 않고 다음 HTTP 요청에 재사용됩니다. 만약 이전 요청 처리 완료 후 `ThreadLocal`에 적재된 테넌트 정보를 명시적으로 제거하지 않으면, 동일한 쓰레드를 재사용하는 다음 다른 사용자의 요청 시 이전 사용자의 Context(테넌트 정보)를 그대로 참조하게 됨을 분석을 통해 확인했습니다.

### 3. Resolution (해결 방안)
*   인터셉터의 `afterCompletion()` 및 서블릿 필터의 `finally` 블록 내에서 `ThreadLocalContextHolder.clear()`를 강제 호출하여, HTTP 요청 생명주기가 끝나는 시점에 해당 쓰레드에 묶인 로컬 데이터를 메모리에서 반드시 반환하도록 공통 격리 로직 구현.
*   또한 주니어들이 임의로 새 스레드를 생성할 때 Context가 누수되는 현상을 막기 위해 사내 ThreadPoolExecutor를 커스터마이징하여, 작업 실행 직전/직후에 자동으로 Context를 인계하고 초기화하는 인터페이스 정립.

### 4. Result (결과)
*   교차 테넌트 간 데이터 침범 및 보안 취약점 원천 차단 (**유출 사례 0건**).
*   안정적인 가비지 컬렉션(GC) 흐름 회복으로 WAS 메모리 가용률 확보.

## 3️⃣ [IoT 수집] 저사양 클라우드 환경 하 IoT 파이프라인 백프레셔(Backpressure) 해결

### 3. Trouble (문제 상황)
*   현장 스마트 헬스케어 디바이스(심박수 측정기 등) 도입 대수가 증가하면서 순간적으로 데이터 송신 빈도가 늘어났을 때, t3.micro(vCPU 2, Memory 1GB)급의 매우 제한된 AWS 가상 환경에서 데이터 수집기(Python Daemon)의 CPU 점유율이 100%를 치며 InfluxDB 적재 지연이 일어났습니다.
*   수집기가 DB에 쓰기 작업을 수행하는 I/O Blocking 시간 동안 대기 큐(Queue)에 센서 패킷이 누적되다 시스템 메모리 초과로 프로세스가 강제 킬(OOM-Killed)되어 데이터가 통째로 유실되는 위기가 있었습니다.

### 2. Deep-Dive & Trade-off (원인 분석 및 대안 검토)
*   **대안 A: 인프라 스케일 업 (Scale-up)**
    *   *장점*: 인스턴스 사양 업그레이드로 가장 빠르고 손쉽게 임시 해결 가능.
    *   *단점*: B2B 솔루션 공급 단가 상승으로 이어져 중소 하청 건설사 고객 유치 경쟁력에 악영향을 줌 (비즈니스적 트레이드오프 부합하지 않음).
*   **대안 B: 비동기 논블로킹 배치(Batch) 적재 아키텍처 개선 (채택)**
    *   *이유*: 시스템 아키텍처 레벨의 최적화를 통해 인프라 비용 추가 없이 소프트웨어 한계 성능을 끌어올리기로 결정.

### 3. Resolution (해결 방안)
*   **단건 쓰기 ➡️ 벌크 배치 쓰기(Bulk Batch Write)**: 매 수집 이벤트마다 InfluxDB API를 호출하던 방식에서, 메모리 내 스레드-세이프(Thread-safe)한 FIFO 버퍼 큐를 두고 **500건 단위 혹은 3초 주기**로 묶어서 벌크 적재하도록 수집 프로세스 변경.
*   **비동기 워커 루프 분리**: Python의 `concurrent.futures` 스레드 풀을 활용하여 MQTT 토픽을 리스닝하고 수신하는 메인 스레드와 InfluxDB 쓰기를 수행하는 워커 스레드의 물리적 생명주기 격리.
*   **Backpressure 제어**: 버퍼 큐가 80% 이상 차오를 경우, 일시적으로 송신 장비 단에 슬립(Sleep) 딜레이를 주는 트래픽 흐름 조율 로직 적용.

### 4. Result (결과)
*   저사양 t3.micro 환경에서 CPU 평균 사용률 **85% ➡️ 25% 이하로 최적화** 성공.
*   장비 확장 시에도 데이터 누락 없는 가용성 **99.9%** 방어.

## 4️⃣ [LocalStorage Sync] 오프라인 버퍼 동기화 시 API 서버 병목 제어

### 4. Trouble (문제 상황)
*   현장에서 지하 공간 진입 등으로 모바일 네트워크가 일시 단절되었다가 복구되는 순간, Browser LocalStorage에 비동기로 누적 쌓여있던 20~30여 개의 미전송 안전 점검 폼이 일시에 백엔드 API 서버로 몰아치는 현상이 목격되었습니다.
*   이로 인해 DB 트랜잭션 락 경합이 생겨 다른 현장 사용자들의 API 요청까지 동반 타임아웃 지연을 겪는 2차 서비스 장애로 번졌습니다.

### 2. Deep-Dive & Trade-off (원인 분석 및 대안 검토)
*   **원인**: 프론트엔드가 온라인 상태 감지 즉시 `Promise.all` 등을 통해 루프를 돌며 모든 보관 데이터를 병렬로 서버 API로 쏘았고, 서버는 단일 클라이언트로부터 들어오는 동시 트랜잭션들을 한꺼번에 처리하려다 커넥션 풀 부족 및 락 대기 현상이 발생했습니다.

### 3. Resolution (해결 방안)
*   **직렬 큐 동기화(Sequential Sync Queue)**: 클라이언트 상태 관리 엔진에서 대기 중인 적재 요청을 병렬이 아닌 직렬 큐 구조로 변환하여, 이전 데이터 전송 API 완료 콜백을 수신한 후 차례대로 다음 데이터를 동기화하도록 흐름 제어.
*   **지수 백오프 및 지터(Exponential Backoff & Jitter)**: 서버가 일시적 부하를 반환(`503 Service Unavailable` 등)할 경우 즉시 재시도하지 않고, 재시도 대기 시간을 점진적으로 늘려가며(1초 ➡️ 2초 ➡️ 4초) 서버로의 충격 완충.

### 4. Result (결과)
*   네트워크 재연결 시의 백엔드 동시 요청 스파이크 부하 **80% 경감**.
*   사용자 입장에서는 폼이 유실되지 않으면서도 백그라운드에서 순차적으로 안전하게 동기화가 완료되는 안정적인 UX 경험 제공.
