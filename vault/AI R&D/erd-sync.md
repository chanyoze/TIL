# erd-sync — ERD ↔ DB 대조·반영 스킬

> ERD 설계 문서(vuerd)와 실제 Oracle DB의 어긋남(drift)을 자동 대조·반영하는 Claude Code 커스텀 스킬. (내부 제품/스키마 식별자 유지, 고객·인명 제거)
>
> **상태: 구현 완료(2026-06-25)** — 아래 [구현 완료](#구현-완료-2026-06-25) 절에 실제 설계·검증 결과 정리. 그 앞 절들은 최초 기획(2026-06-19)이며, 일부는 구현하며 바뀌었다.

## 배경 / 동기

`ref-erd\<카테고리>\*.vuerd.json` 형태로 ERD(ERD-Editor / vuerd v3) 설계 문서를 관리한다. 실제 운영 DB(SKPMIS / PMIS4, Oracle, readonly MCP)와 시간이 지나며 어긋난다(drift). 문서관리(DCMT) ERD를 SKPMIS DB와 **수동으로 대조**해 테이블 누락·컬럼 차이·타입/NOT NULL 차이를 찾아냈고, 이 반복 작업을 **스킬로 자동화**한다.

## 무엇을 할지

ERD 파일 1개와 대상 DB를 받아서:

1. vuerd(v3) 파싱 → 테이블/컬럼/타입/NotNull/PK 추출
2. Oracle 데이터 딕셔너리(`all_tab_columns`·`all_col_comments` 등) 조회
3. 둘을 대조해 차이를 **6가지로 분류**
4. **🟢자동반영 가능 / 🟡사람 판단 필요**로 나눠 제시
5. 승인된 항목을 `vuerd.json`에 **직접 반영**(백업 후)
6. DB가 빠진 항목은 `migration_*.sql`(ALTER/CREATE)로 **역생성** — MCP가 readonly라 DB 직접 변경 불가
7. 표준 포맷으로 git 커밋

### 차이 분류 6종

| # | 유형 | 예 (문서관리 대조) |
|---|------|------------------|
| 1 | 테이블 누락 | ERD엔 있고 DB엔 없는 테이블 |
| 2 | DB엔 있고 ERD에 빠진 컬럼 | `UPD_PSBL_TF`, `*_CYCLE.REM` 등 |
| 3 | 컬럼명 변경(rename) 추정 | `DEL_PSBL_TF` → `DOC_DEL_PSBL_TF` |
| 4 | ERD엔 있고 DB에 없는 컬럼 | `CYC_SN`, `CFM_USE_TF` 등 |
| 5 | 데이터타입 차이 | `VARCHAR2(10)`→`(30)`, `(500)`→`(4000)` |
| 6 | NOT NULL 제약 차이 | ERD NN / DB NULL, 또는 그 반대 |

### 자동반영 vs 판단필요 정책

| 🟢 자동반영 (DB→ERD 문서 교정) | 🟡 반드시 질문 |
|---|---|
| 컬럼 추가, 타입/길이 갱신, rename(코멘트·위치 매칭), stale 잔재 컬럼 삭제 | 테이블 통째 누락, DB에 없는 ERD 전용 컬럼/제약, NN 강화 |

핵심 원칙: DB가 운영 기준이지만 **NN 누락·테이블 누락은 DB 결함일 수 있어** ERD에 무조건 덮어쓰지 않는다. 이 분기를 스킬이 강제한다.

## 어떤 기능

### MVP (1차)
- vuerd v3 파싱 → 표준 JSON
- DB 딕셔너리 대조 → 6분류 리포트(표)
- **자동반영**: 컬럼 추가 / 타입·길이 갱신 / rename
- **백업 + dry-run preview**(변경 전/후 diff 확인 후 확정)
- 🟡 판단필요 건 한 건씩 사용자 확인
- DB 결함 건 → `migration_<db>.sql` 역생성
- 표준 커밋(날짜/이름/요청자/요약)

### 이후 확장 (2차+)
- PK / FK / 인덱스 / 컬럼순서 / 코멘트 텍스트까지 대조
- `ref-erd` **전체 카테고리 일괄 드리프트 대시보드**
- 타임스탬프 드리프트 스냅샷(변화 추적)
- **alias-dict 연동**: 새 컬럼 한글 코멘트 → 영문 별칭·명명규칙 린트
- **writedown 연동**: 대조 리포트를 raw/readable로 기록

## 간단한 구성 (폴더 구조)

```
~/.claude/skills/erd-sync/
├── SKILL.md          ← 트리거·절차 (기존 스킬과 동일 위치/형식)
└── scripts/          ← 결정적 로직 (매번 즉석 코드 대신 검증된 스크립트)
    ├── parse_erd.js  ← vuerd v3 → 표준 JSON
    ├── diff.js       ← ERD vs DB 비교 → 6분류 + 자동/질문 태깅
    ├── apply.js      ← ★핵심: vuerd 변형(컬럼 추가/타입/rename/삭제) + 백업
    └── gen_ddl.js    ← DB 결함 건 → migration SQL 역생성
```

- 기존 커스텀 스킬(writedown·alias-dict)은 `SKILL.md` 한 장이지만, erd-sync는 파싱·변형 로직이 많아 `scripts/`로 분리한다(안정성).
- **핵심 난이도 = `apply.js`**: vuerd v3는 `tableColumnEntities`(ID)·`columnIds` 배열·UI 좌표·`relationshipEntities`·`indexColumnEntities`가 얽혀 있어, 컬럼 추가/삭제 시 참조를 일관되게 갱신해야 깨지지 않는다. 이 변형 헬퍼가 견고하면 전 카테고리에 재사용된다.

## 사용 방법

```
/erd-sync <ERD파일경로 또는 카테고리폴더> <DB: skpmis|pmis4>
```

예) `/erd-sync "ref-erd\340. 문서관리 (DCMT)" skpmis`

실행 시 자동 흐름:

1. ERD 파싱 + DB 딕셔너리 조회
2. 6분류 대조 리포트 출력(표)
3. 🟢 자동반영 후보 + 🟡 판단필요 건 제시
4. 🟡 건은 한 건씩 확인 받음
5. 승인분 → `.vuerd.json` 백업(`.bak`) 후 반영, 변경 diff 미리보기
6. DB 결함 건 → `migration_<db>.sql` 생성
7. 표준 포맷으로 커밋

## 만드는 순서

1. `parse_erd.js` (덤프 로직 재사용 — 거의 완성)
2. `diff.js` (수동 대조 스크립트 발전형)
3. `apply.js` ← 가장 공들일 곳, **문서관리 ERD로 즉시 실전 검증**
4. `gen_ddl.js` + `SKILL.md`
5. 찾은 문서관리 수정사항을 이 스킬로 첫 반영(= 스킬 검증 + 원래 목적 동시 달성)

## 제약 / 메모

- Oracle MCP는 **readonly** 모드 → DB 직접 ALTER 불가. DB 변경이 필요한 건은 SQL로 역생성만.
- MCP 끊김 대비: `MCP_TIMEOUT=60000`(settings.json) 적용해 오라클 연결 안정화 완료(2026-06-19).
- 수동 대조 범위: 컬럼·타입·NOT NULL만. PK/FK·인덱스·컬럼순서·코멘트는 2차에서.

## 구현 완료 (2026-06-25)

PoC로 타당성을 먼저 확인한 뒤 스킬로 만들고, 한 도메인(테이블 25·FK 15)으로 회귀검증까지 끝냈다. 구현하면서 기획과 달라진 부분이 꽤 있다.

### 최종 구조 — 2단계 (Plan → Apply)

기획의 단일 흐름이, **범위 지정**과 **부분 포함** 요구 때문에 두 단계로 갈렸다.

```
[준비] 접두어 패턴(예: TXXX%) + 대상 폴더
   │
   ▼ PHASE 1 — 계획
  DB 스캔 → plan.md 자동 분류(full / partial / exclude)
   │        ↑ ❶ 사람이 검토(포함여부·표시컬럼)
   ▼ PHASE 2 — 적용
  sync 또는 bootstrap → report.md(변경분)
   │        ↑ ❷ 사람이 리포트 확인 → OK
   ▼
  *.synced.vuerd.json (참조 무결성 통과해야 출력)
   │        ↑ ❸ erd-editor로 열어 육안 확인 (최종)
   ▼ 완료
```

확인 게이트는 **❶계획 ❷리포트 ❸시각** 셋뿐. 컬럼·관계 수기입력은 0.

### 테이블 포함 3등급 (신설)

기획에 없던, 가장 실용적인 추가. 거대한 참조 테이블을 통째로 그리지 않게 해준다.

| 등급 | 의미 | 표시 컬럼 |
|---|---|---|
| **full** | 도메인 핵심 테이블 | 전체 컬럼 |
| **partial** | FK로만 엮인 거대 참조 테이블 | **PK + 다른 테이블이 참조하는 컬럼 + 사용자 지정만** |
| **exclude** | 범위 밖 | 미포함(관계는 리포트에만) |

> 예: 컬럼이 100개에 옵션 기능까지 달린 참조 테이블이라도, ERD엔 PK와 실제 참조되는 몇 개만 노출 → 관계만 깔끔히 표현. (drift = ERD와 실제 DB 스키마의 어긋남)

### 두 가지 모드

- **sync**: 기존 vuerd가 있을 때. 테이블 **좌표·색상·메모를 100% 보존**하고 내용만 DB 기준으로 갱신.
- **bootstrap**: ERD가 없는 신규 도메인. DB 메타만으로 격자 배치 ERD를 새로 생성.

### 기획 대비 바뀐 점

- 단일 흐름 → **2단계(계획/적용)**.
- **관계(FK)·코멘트 대조를 1차에 포함**(기획은 2차 예정이었음). 코멘트는 DB 우선으로 덮되 전부 리포트로 추적.
- **migration DDL 역생성은 1차에서 보류** — 동기화/생성·부분포함·무결성 검증에 집중.
- **무결성 가드**: 참조 무결성(테이블·컬럼·관계 ID 상호참조, 표시순서 일관성) 검증을 통과 못하면 **파일을 아예 안 쓴다**.
- **자동 정리**: 원본에 누적돼 있던 미참조(죽은) 컬럼·끊긴 인덱스를 정리.

### 검증 결과

한 도메인으로 세 모드 모두 참조 무결성 통과:

| 모드 | 결과 |
|---|---|
| sync(회귀) | 컬럼 추가 7·제거 4·변경 37·관계 +1, **좌표 0 이동**(레이아웃 완전 보존) |
| partial | 참조 테이블을 key 1개로 축소하면서 관계는 유지 |
| bootstrap | DB만으로 25테이블·14관계·349컬럼 격자 생성 |

대조 과정에서 타입 누락(공백→Number), VARCHAR2 길이 변경, NOT NULL 다수 차이를 잡아냈고, **DB 코멘트 쪽 오타**까지 드러났다(DB 우선이라 덮이지만 리포트로 사람이 확인하는 구조라 안전).

### 배운 점 / 한계

- **"완전 자동"은 아니지만 노동의 종류가 다르다** — 사라지는 건 컬럼·관계 수백 개 수기입력이고, 남는 건 굵직한 확인 3번. 게다가 DB가 늘 옳은 건 아니라(오타·결함) 사람 게이트는 한계라기보다 안전장치다.
- **마지막 1마일은 사람 몫** — JSON 구조·참조 무결성은 자동 검증하지만, 실제 렌더링은 헤드리스로 못 봐서 사람이 1회 열어 확인해야 한다. 정직한 경계.
- vuerd v3는 엔티티가 ID로 얽혀 있어(컬럼·관계·인덱스 상호참조), 변형 후 **참조 무결성 검사를 반드시 깔아야** 조용히 깨진 파일이 안 나온다.
