## 프롬포트 엔지니어링

### 개념
AI 모델이 우리의 의도를 정확하게 이해하고 원하는 결과를 생성하도록 입력 지시사항을 최적화하는 기술

### 주요 기법 (2~4가지)

#### 1. **Chain of Thought (CoT)**
- **정의**: AI가 단계별로 사고 과정을 거쳐 답을 도출하도록 유도하는 기법
- **예시**: 복잡한 문제를 풀 때 "먼저 X를 고려하고, 그 다음 Y를 계산한 후, 마지막으로 Z를 결정하세요"라고 지시
- **효과**: 정확성 향상, 특히 수학, 논리, 추론이 필요한 작업에서 효과적
- **한계**: 처리 시간 증가, 토큰 사용량 증가

#### 2. **Role Prompting (역할 기반 프롬포트)**
- **정의**: AI에게 특정 역할(예: 전문가, 멘토, 코드 리뷰어)을 부여하는 기법
- **예시**: "당신은 20년 경력의 풀스택 개발자입니다. 이 코드를 리뷰해주세요"
- **주의점**: 미래의 AI는 점진적으로 간소화되어 최소한의 명시 없이도 문맥을 이해할 가능성 높음
- **현재 활용**: 안전장치 역할, 결과의 일관성 유지를 위해 사용하되 과도한 역할 설정은 지양

#### 3. **Few-Shot Learning**
- **정의**: 문제를 풀기 전에 몇 가지 예시(샘플)를 제공하여 패턴을 학습하도록 하는 기법
- **예시**: 
  ```
  예시 1) 입력: "안녕하세요" → 출력: "공손함"
  예시 2) 입력: "뭐야" → 출력: "무례함"
  이제 분석: "이건 뭐지?"
  ```
- **효과**: Zero-shot 학습보다 정확도 높음, 특정 스타일이나 포맷 유지에 효과적

### 프롬포트 엔지니어링의 한계

| 한계 | 설명 |
|------|------|
| **수동 작업** | 매번 프롬포트를 직접 작성하고 조정해야 함 |
| **복잡한 작업** | 여러 단계가 필요한 작업에는 효과 제한적 |
| **결과 후처리** | 코드 작성, 데이터 수정 등 추가 작업 필요 |
| **확장성 부족** | 프롬포트 길이 증가 시 성능 저하 가능 |

### 프롬포트 엔지니어링의 미래
점진적으로 더 간단한 요청만으로도 AI가 충분히 이해하고 수행할 수 있는 방향으로 발전할 것으로 예상

## 컨텍스트 엔지니어링

### 핵심 개념
**Context Window에 가용한 메모리의 한계 내에서, 필요한 정보만 정확하게 전달하여 최대 효율을 내는 기술**

### Context Window란?
- AI 모델이 한 번에 처리할 수 있는 최대 토큰(단어) 수
- 예: Claude의 Context Window는 200K 토큰 (대략 150,000 단어)
- 초과하면 정보 손실 또는 성능 저하 발생

### 컨텍스트 오염 (Context Pollution)
```
문제: 과도한 정보 입력 → 데이터 오염 증가 → 결과 정확도 하락

예시:
❌ 200개 파일 내용 전부 + 100개 문서 + 전체 채팅 히스토리
✅ 실제 필요한 3개 파일 + 관련 문서 2개 + 최근 10개 메시지
```

### 컨텍스트 엔지니어링 전략

#### 1. **정보 필터링**
- 작업에 필수적인 정보만 추출
- 불필요한 세부사항 제거
- 우선순위 기반 선택

#### 2. **정보 구조화**
- 마크다운 포맷 활용
- 섹션/카테고리별 구분
- 명확한 계층 구조

#### 3. **요약 및 압축**
- 긴 내용은 요약본 사용
- 핵심 포인트만 추출
- 필요시 링크/참조로 대체

### 컨텍스트 엔지니어링의 한계

| 한계 | 설명 | 해결책 |
|------|------|--------|
| **반복적 수동 트리거** | 매번 사람이 작업 시작 신호 제공 필요 | 자동화 도구 (n8n, Zapier 등) |
| **도구 수동 연결** | 각 Tool을 일일이 연결하는 과정 필요 | MCP(Model Context Protocol) 활용 |
| **학습 부재** | AI가 자동으로 도구/내용 학습 불가 | 에이전틱 엔지니어링 도입 |

### MCP (Model Context Protocol)
- **역할**: 다양한 도구/서비스를 AI와 연결하는 표준 프로토콜
- **장점**: 도구 연결 간소화, 자동 업데이트 지원
- **활용**: Claude가 직접 파일 시스템, API, 데이터베이스 접근 가능

## 워크플로우 자동화

### 개념
반복적인 작업을 자동화 도구로 연결하여 수동 개입 없이 처리하는 기술

### 주요 도구

#### **n8n (엔에이트엔)**
- **정의**: 오픈소스 워크플로우 자동화 플랫폼
- **활용 예시**: 건설안전 전문가 챗봇
  - 사용자 입력 → 데이터베이스 조회 → AI 처리 → 응답 생성 → 로그 저장
- **특징**: 
  - 코딩 없이 노드 기반으로 워크플로우 구성
  - 200+ 통합 앱 지원
  - 조건문, 루프 등 복잡한 로직 가능

#### **다른 자동화 도구**
- **Zapier**: 클라우드 기반, SaaS에 특화
- **Make**: 고급 자동화, 복잡한 시나리오
- **AutoMate**: 데스크톱 자동화

### 워크플로우 자동화의 장점
✅ 수동 작업 시간 단축  
✅ 인적 오류 감소  
✅ 24/7 자동 운영  
✅ 확장성 용이  

### 오케스트레이션 (Orchestration)
- **정의**: 여러 개의 자동화 작업을 조합하여 복잡한 프로세스 관리
- **예**: 주문 접수 → 재고 확인 → 배송사 연동 → 고객 알림
- **필요성**: 단순 자동화로는 해결 불가능한 비즈니스 프로세스 구현

## 에이전틱 엔지니어링

### 핵심 개념
**AI 에이전트가 자동으로 목표 달성을 위해 필요한 모든 것을 계획하고, 실행하고, 평가하는 기술**

### 에이전틱 AI의 3대 구성요소

#### 1. **PLAN (계획)**
- **역할**: 주어진 목표를 작은 단계(sub-task)로 분해
- **예시**: 
  - 목표: "Python으로 데이터 분석 대시보드 만들기"
  - 분해: 
    1. 데이터 수집
    2. 데이터 정제
    3. 가시화 라이브러리 선택
    4. UI 작성
    5. 배포
- **방식**: Tree of Thought, Chain of Thought 등 기법 활용

#### 2. **ACT (행동)**
- **역할**: 계획된 단계를 실제로 실행
- **가능한 행동**:
  - 💻 코드 실행 (Python, JavaScript 등)
  - 🔍 웹 검색 (최신 정보 수집)
  - 📁 파일 조작 (읽기, 쓰기, 삭제)
  - 🌐 API 호출 (외부 서비스 연동)
  - 🗣️ 사용자 상담 (명확화 필요시)

#### 3. **REFLECT (반영)**
- **역할**: 행동 결과 평가 후 다음 단계 결정
- **프로세스**:
  1. 결과 검증 (성공/실패)
  2. 오류 분석 (실패 원인 파악)
  3. 계획 수정 (필요시 재계획)
  4. 다음 행동 결정
- **중요성**: 무한 루프 방지, 효율적 자원 사용

### 에이전틱 엔지니어링의 3대 새로운 문제

#### ❌ **1. 컨텍스트 불안정성**
- **문제**: Context Window 한계에 도달하면 갑자기 작업 종료
- **증상**: 장시간 작업 중 메모리 부족으로 갑자기 멈춤
- **영향**: 진행 중인 작업 손실, 사용자 경험 저하
- **해결책**:
  - 주기적인 체크포인트 저장
  - 오래된 컨텍스트 정리
  - 작업 분할 처리

#### ❌ **2. 자기평가 편향 (Self-Evaluation Bias)**
- **문제**: AI가 자신의 결과를 객관적으로 평가하지 못함
- **예시**: Claude가 작성한 코드를 Claude가 검증 → 실제 오류 놓칠 수 있음
- **해결책**:
  - 다른 모델로 검증 (예: Claude → CodeX → Claude)
  - 외부 테스트 도구 활용
  - 사람의 리뷰 단계 추가

#### ❌ **3. 장시간 작업 불일치**
- **문제**: 세션이 길어지면 초반 지시사항과 후반 행동이 모순
- **원인**: Context Window에서 초기 정보 손실, 지시사항 희석
- **증상**: "아까 말씀하신 것과 다르게 했어요"
- **해결책**:
  - 핵심 지시사항을 시스템 프롬포트로 고정
  - 정기적으로 목표 재확인
  - 작업 로그 유지

## 하네스 엔지니어링

### 개념
AI의 행동을 제어하고 제약하기 위해 명확한 규칙과 가이드를 설정하는 기술  
**"AI의 자유도를 적절히 제한하면서 효율성을 극대화하는 방법"**

### 주요 구성 요소

#### **1. claude.md (가이드라인 문서)**
- **역할**: 프로젝트의 전반적인 방향, 스타일, 관례 명시
- **특징**: 
  - 부탁에 가까운 성격 (강제성 낮음)
  - AI가 참고만 하고 어기기 쉬움
- **작성 내용**:
  - 프로젝트 목표 및 개요
  - 코드 스타일 가이드
  - 기술 스택 및 아키텍처
  - 네이밍 컨벤션
  - 문서화 표준
- **예시**:
  ```markdown
  # 프로젝트 가이드
  - 변수명은 camelCase 사용
  - 함수는 동사로 시작
  - 모든 public 함수는 주석 필수
  ```

#### **2. Hook (강제성 규칙)**
- **역할**: AI가 반드시 따라야 할 규칙 (Skill.md에 명시)
- **특징**: 
  - 강제성 높음
  - 시스템 수준의 제약
  - 위반 시 에러 발생
- **활용 예시**:
  ```
  ❌ Hook) "절대로 깃헙에 자동으로 push 하지 말 것"
  ❌ Hook) "데이터베이스 직접 삭제 금지"
  ✅ Hook) "코드 실행 전 항상 사용자 확인 필수"
  ```

### claude.md vs Hook 비교

| 항목 | claude.md | Hook |
|------|-----------|------|
| **강제성** | 낮음 (부탁) | 높음 (의무) |
| **위반 시** | 참고만 하지 않음 | 실행 차단 |
| **변경 빈도** | 자주 (동적) | 드물게 (정적) |
| **저장 위치** | 프로젝트 폴더 | Skill.md |
| **예시** | "가급적 TypeScript 사용" | "반드시 테스트 먼저" |

### claude.md 작성 스코프

#### **1. Global (전역 스코프)**
- 모든 프로젝트에 적용되는 회사 정책
- 예: "모든 API 응답은 ISO 8601 날짜 포맷"
- 위치: 회사 공유 문서

#### **2. Project (프로젝트 스코프)**
- 팀 단위 또는 특정 프로젝트에만 적용
- 예: "이 프로젝트는 React + TypeScript 기반"
- 위치: 프로젝트 루트의 claude.md

#### **3. Local (로컬 스코프)**
- 개인 개발 환경에만 적용
- 예: "내 머신에서는 Docker로 실행"
- 위치: .env, .claude (미포함 파일)

### claude.md 작성 지침

#### ✅ **필수 원칙**
1. **구체성**: 모호한 표현 지양, 명확한 요구사항 명시
   ```
   ❌ 나쁜 예: "좋은 코드 작성"
   ✅ 좋은 예: "함수 길이 50줄 이하, 순환 복잡도 3 이하"
   ```

2. **팀 전체 유효성**: 한 명만 해석 가능한 내용 제외
   ```
   ❌ "내가 좋아하는 스타일로"
   ✅ "ESLint 설정 airbnb 기반 사용"
   ```

3. **정적 내용만**: 자주 변하는 내용은 claude.md에 비추천
   ```
   ❌ "현재 환경변수는 다음과 같습니다" (자주 변함)
   ✅ "env 파일을 .gitignore에 추가" (불변)
   ```

4. **200줄 이하 유지** (필수)
   - 너무 길면 AI가 모두 읽지 않음
   - 최소한의 필수 정보만 포함
   - 세부사항은 기술 문서로 분리

#### 📏 **구조 예시**
```markdown
# Project Claude.md
## 프로젝트 개요
- 목적: RESTful API 서버
- 기술 스택: Node.js, Express, PostgreSQL

## 개발 규칙
- 언어: TypeScript
- 코드 스타일: ESLint (airbnb)
- 브랜치: main/develop/feature/*
- 커밋: Conventional Commits

## 제약사항
- 데이터베이스 마이그레이션은 수동 승인 필수
- 프로덕션 환경에 직접 접근 금지
```

## Skills & Hooks 심화

### Skills의 역할 (자동화)

#### **기본 개념**
프롬포트마다 "이 파일을 읽어줘, 저 작업을 해줘"라고 반복할 필요 없음  
→ Skill.md에 미리 정의하면 자동 트리거

#### **Skills의 구조**
```
skill-name/
├── SKILL.md          # 스킬 정의서
├── examples/
└── tests/
```

#### **실제 예시: /deep-interview 스킬**
- **목적**: 사용자와 대화를 통해 개발할 내용 점진적으로 좁혀나가기
- **동작 방식**:
  1. 사용자 요구사항 청취
  2. 핵심 질문으로 필요사항 정제
  3. 기술 스택 결정
  4. 개발 계획 수립
- **장점**: 모호한 요청 → 명확한 스펙으로 변환

### 민감한 정보 처리

#### **문제 상황**
AI가 개인정보, API 키, 보안 정보 필요할 때 자연스러운 처리 방법?

#### **해결책**
```
1. env 파일 참조
   - 보안 정보는 .env에 저장
   - AI는 필요시만 읽도록 권한 설정

2. 자연어 요청
   - "당신의 API 키를 입력하세요"
   - 사용자가 직접 입력 후 진행

   - 특정 파일만 읽을 수 있도록 제한
   - 쓰기 권한은 더욱 제한적
```

## Generator-Evaluator 패턴

### 개념
하나의 AI 모델이 생성(Generate)하고, 다른 AI 모델이 평가(Evaluate)하는 이중 검증 시스템

### 흐름
```
Generator (생성)
    ↓
산출물 (코드, 텍스트, 이미지)
    ↓
Evaluator (평가)
    ↓
피드백
    ↓
Generator (수정)
```

### 예시
```
1. Claude Sonnet (빠름, 저비용)
   → 코드 생성, 문서 작성

2. Claude Opus (강력함, 고비용)
   → 생성된 결과 검증, 오류 지적

3. Claude Haiku (가장 빠름, 최저비용)
   → 형식 검증, 기본 문법 확인
```

### 장점
✅ 검증 단계 자동화  
✅ 오류 최소화  
✅ 비용 효율화 (빠른 모델로 생성, 강력한 모델로 검증)  
✅ 품질 향상

## 모델 선택 전략

### Claude 모델 라인업

| 모델 | 비용 | 속도 | 능력 | 주용도 |
|------|------|------|------|--------|
| **Haiku** | 💰 매우 낮음 | ⚡⚡⚡⚡⚡ 매우 빠름 | ⭐⭐ | 간단한 작업, 형식 검증 |
| **Sonnet** | 💰💰 중간 | ⚡⚡⚡ 중간 | ⭐⭐⭐⭐ | 메인 에이전트, 균형잡힌 선택 |
| **Opus** | 💰💰💰 높음 | ⚡⚡ 느림 | ⭐⭐⭐⭐⭐ | 복잡한 추론, 검증, 품질 검사 |

### 계층적 에이전트 구조

#### **Multi-Agent 아키텍처**
```
사용자 요청
    ↓
메인 에이전트 (Sonnet)
    ├─ Haiku: 빠른 전처리 (데이터 정제)
    ├─ Sonnet: 메인 로직 처리
    └─ Opus: 최종 검증 및 추론
    ↓
최종 결과 반환
```

#### **선택 기준**
- **속도 중요**: Haiku
- **비용 효율**: Haiku + Sonnet 조합
- **품질 중요**: Sonnet + Opus 조합
- **정확성 최우선**: Opus로 최종 검증
