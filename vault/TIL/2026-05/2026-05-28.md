## Google Sheets MCP 연동 (a-bonus/google-docs-mcp)

**배운 것**
Claude Code에 Google Sheets(+Docs/Drive/Gmail/Calendar) MCP 서버를 붙이려면 GCP 프로젝트 생성 → API 활성화 → OAuth 설정 → 토큰 발급 → MCP 등록 순서로 6단계가 필요하다.

**핵심 내용**
- MCP(Model Context Protocol)는 Claude가 외부 도구/데이터에 접근하게 해주는 표준 프로토콜
- `a-bonus/google-docs-mcp`는 Google Workspace 전체(Sheets, Docs, Drive, Gmail, Calendar)를 한 번에 다룰 수 있는 Node.js 기반 MCP 서버
- 인증 방식: OAuth Client ID(Desktop App) → refresh token을 로컬에 저장
- 토큰 저장 위치: `C:\Users\lch\.config\google-docs-mcp\token.json`
- MCP 등록 스코프 3가지: `user`(모든 프로젝트, 본인만) / `local`(현재 프로젝트) / `project`(.mcp.json 커밋, 팀 공유)

**예시 / 코드**

### STEP 1 — GCP 프로젝트 생성
1. https://console.cloud.google.com 접속
2. 상단 프로젝트 셀렉터 → **새 프로젝트(NEW PROJECT)**
3. 이름 예: `claude-mcp` → 만들기

### STEP 2 — API 5개 활성화
**API 및 서비스 > 사용 설정된 API > + API 및 서비스 사용 설정** 에서 검색·활성화:
- Google **Docs** API
- Google **Sheets** API
- Google **Drive** API
- **Gmail** API
- Google **Calendar** API

### STEP 3 — OAuth 동의 화면 설정
**API 및 서비스 > OAuth 동의 화면**
1. User Type: **외부(External)** → 만들기
2. 앱 이름 / 사용자 지원 이메일 / 개발자 연락처 입력
3. 범위(Scopes) 단계는 비워두고 통과
4. **테스트 사용자**에 본인 이메일 추가 (필수)
5. 저장

### STEP 4 — OAuth Client ID 발급
**API 및 서비스 > 사용자 인증 정보 > + 사용자 인증 정보 만들기 > OAuth 클라이언트 ID**
- 애플리케이션 유형: **데스크톱 앱(Desktop app)**
- 생성 후 뜨는 **Client ID** / **Client Secret** 안전하게 보관

### STEP 5 — Refresh Token 발급 (PowerShell)
```powershell
$env:GOOGLE_CLIENT_ID="여기에_client_id"
$env:GOOGLE_CLIENT_SECRET="여기에_client_secret"
npx -y @a-bonus/google-docs-mcp auth
```
- 브라우저 열림 → Google 로그인 → 권한 동의
- 성공 시 `C:\Users\lch\.config\google-docs-mcp\token.json` 생성
- "확인되지 않은 앱" 경고는 **고급 → 안전하지 않은 페이지로 이동** 클릭

### STEP 6 — Claude Code에 MCP 등록 (user 스코프)
```powershell
claude mcp add google-docs -s user `
  -e GOOGLE_CLIENT_ID=여기에_client_id `
  -e GOOGLE_CLIENT_SECRET=여기에_client_secret `
  -- npx -y @a-bonus/google-docs-mcp
```

확인:
```powershell
claude mcp list
```
Claude Code 재시작 후 `/mcp` 입력 → `google-docs` connected 상태 + 툴 목록 확인.

### 동작 테스트 예시
- "내 Google Drive에서 스프레드시트 목록 보여줘"
- "새 시트 만들어서 A1에 'hello' 적어줘"

**더 알아볼 것**
- Service Account 방식(`SERVICE_ACCOUNT_PATH`, `GOOGLE_IMPERSONATE_USER`)으로 전환 — 엔터프라이즈/도메인 위임용
- `mcp-gsheets`(Python/uvx) 대안 — Sheets만 가볍게 쓸 때
- 토큰 만료 / 재인증 절차 (`token.json` 삭제 후 `auth` 재실행)
- 시트 자동화 스킬 만들기 (단어장 동기화, TIL 시트화 등)

## Oracle DB MCP 연결 (SQLcl 실패 → mcp-oracle-db 채택)

**배운 것**
사내 Oracle DB(dev, 19c)를 Claude Code에 MCP로 붙이려고 Oracle 공식 경로(SQLcl)를 먼저 시도했으나 이 PC 환경의 ACL 버그로 막혔고, node-oracledb thin 모드 기반 커뮤니티 패키지 `mcp-oracle-db`로 우회해 성공했다. DB 접속 방법은 크게 두 갈래(공식 SQLcl / 드라이버 직접접속)가 있다는 걸 배움.

**핵심 내용**
- **MCP 서버 = Claude와 외부 시스템(DB) 사이의 중개 프로그램.** 붙으면 자연어로 실제 테이블 조회 가능.
- Oracle에 붙는 두 갈래: ① **SQLcl `-mcp`**(공식, 저장된 named connection 기반) ② **node-oracledb 드라이버 직접 접속**(커뮤니티 패키지).
- 새 MCP 도구(`mcp__oracle__*`)는 **Claude Code 세션 재시작 후** 로드된다. `claude mcp list`로 연결 상태(✓ Connected)만 먼저 확인 가능.
- node-oracledb **thin 모드**는 Oracle Instant Client 설치 없이 순수 JS로 접속 → 설치 부담 없음. 6.x 기본값이 thin.

### 1안 — SQLcl 공식 MCP (실패)
**시도**: SQLcl 26.1 다운로드·설치 → `connect -save`로 접속 저장 → `sql -mcp` 등록 계획.
**결과**: DB 접속 자체는 성공(PMIS4로 19c 접속 OK)하나 **접속 저장(`-save`)이 항상 실패.**
**원인(핵심)**: SQLcl이 접속 저장 폴더(`%APPDATA%\DBTools\connections`)의 ACL을 보안처리하면서, **현재 사용자 SID에서 끝 RID(`-1001`)를 누락한 잘린 SID**(`S-1-5-21-...-2174127946`)에게만 권한을 줌 → **자기 자신을 폴더에서 잠가버려** 이후 읽기/쓰기가 `AccessDeniedException`.
- 빈 디렉터리에서도 재현, **SQLcl 26.1·25.2 모두 동일**, JDK 21/25 교체해도 동일 → 버전·JDK 문제 아닌 **환경 고유 버그**로 결론.
- 매 실행마다 폴더를 다시 보호(잠금)하므로 ACL을 수동으로 고쳐도 무의미 → SQLcl 경로 포기.
- 참고: Oracle 공식 저장소 `github.com/oracle/mcp`의 `dbtools-mcp-server`도 같은 DBTools 저장소 사용 → 동일하게 막힘.

### 2안 — mcp-oracle-db (채택, 성공)
**패키지**: `mcp-oracle-db` (npm, node-oracledb thin 모드).
**등록**(user 범위, 읽기전용):
```powershell
claude mcp add oracle -s user `
  -e ORACLE_USER=PMIS4 `
  -e ORACLE_PASSWORD=*** `
  -e ORACLE_CONNECTION_STRING=호스트:1521/서비스명 `
  -e ORACLE_MODE=readonly `
  -- npx -y mcp-oracle-db
```
- 접속 형식: `host:1521/service` (TNS alias·Cloud wallet도 지원).
- `ORACLE_MODE` 기본 `readonly`(안전). 쓰기 필요 시 `readwrite` → `execute`(DML)/`ddl` 도구 활성화.
- 노출 도구: `list_schemas`, `list_tables`, `describe_table`, `query`(SELECT), `explain_plan`, `session_info`, `active_sessions`, `locks`.
- **검증**: `claude mcp list` → ✓ Connected. thin 모드 직접 SELECT로 `user_tables` 2357개 확인.

**대안(2안)의 잠재 리스크**
- **서드파티 코드 신뢰**: 공식이 아닌 커뮤니티 패키지에 DB 자격증명(env)을 넘긴다. 로컬·dev 한정이라 수용했지만, 운영 DB·팀 배포 시엔 재검토 필요(코드 감사 또는 자체 제작).
- **유지보수 불확실성**: 패키지 업데이트 중단·호환성 깨짐 가능. `npx -y`는 매번 최신을 끌어올 수 있어 버전 고정(특정 버전 지정) 고려.
- **자격증명 평문 저장**: 접속정보가 `~/.claude.json`(user 설정)에 평문으로 들어감. 공유 PC면 주의.
- **thin 모드 기능 한계**: 일부 고급 기능(특정 인증/Advanced Queuing 등)은 thick 모드 필요할 수 있음. 현재 조회 용도엔 무관.

**기타 주요 사항**
- 디트어 산출물(SQLcl 26.1/25.2, JDK 21, 각종 zip·home 폴더)은 `C:\tools\`에서 전부 정리(삭제)함. 일부 폴더/파일은 SQLcl이 남긴 깨진 ACL 때문에 `takeown`+`icacls`로 소유권 회수 후 삭제.
- 메모리에 기록: 다음에 Oracle MCP 이슈 재발 시 SQLcl 재시도하지 말고 `mcp-oracle-db`로 갈 것.

**더 알아볼 것**
- 팀 배포 시 자체 최소 MCP 서버 제작(node-oracledb thin, 코드 통제) — 서드파티 신뢰 리스크 제거.
- `npx -y` 대신 버전 고정 또는 로컬 설치로 재현성 확보.
- `readwrite` 모드 운영 정책(누가/언제 쓰기 허용할지).
- SQLcl ACL 버그 원인 추적(왜 RID가 누락되는지) — Oracle 버그 리포트 가치.
