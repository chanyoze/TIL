# 4. 목표별 진단 & 단계별 가이드

당신이 말한 소기의 목적:
**(A) top + 게시판 2~3개 디자인 리팩토링**, **(B) 향후 React·Nuxt.js·PostgreSQL 전환 문서화.**

아래는 "지금 당장 할 일(A)"과 "미래 설계(B)"를 분리한 로드맵입니다.

---

## Phase 0 — 빌드/실행 확인 ✅ 완료 (2026-06-02)
> 무엇이든 바꾸기 전에 **현재 화면이 뜨는지** 기준점을 잡았습니다.

**실행 결과:**
- 빌드: `cloudlab-wt-core:2.3.1`이 로컬 `.m2`에 캐시돼 있어 **컴파일 성공**. (단, `spring-boot:run` 플러그인 의존성은 오프라인에 없어 **온라인 모드 필요**. GitHub 인증 토큰은 cloudlab 캐시 덕에 불필요했음.)
- 환경: Java 25 런타임, 컴파일 타깃 `release 21`. 실행 명령:
  `./mvnw -s settings.xml spring-boot:run -Dspring-boot.run.profiles=local`
- 기동: **포트 10040에서 정상 기동(약 47초)**. DB 접속 실패 없이 부팅됨(HikariCP lazy).
- 스모크 테스트(전부 **HTTP 200**): `/`, `/board/news`, `/board/news_detail`, `/company/greeting`, `/solution/cleverlink`.
- 페이지 title은 `envVariables.title` → `[Local] CIP SYSTEM 씨아이피시스템`로 정상 바인딩.

**확인된 사실(분석 검증):**
- 기동 로그에 `No MyBatis mapper was found in '[com.cloudlab.wt.**.mapper]'` → **매퍼 부재 확정**.
- `/board/news` 응답 HTML에 `<%= ... %>` 잔재가 **5건 그대로 브라우저로 전송됨** → 죽은 ASP 코드가 실제 노출 중(Phase 2에서 제거 대상).

**산출물:** 실행 재현 명령 확보 + 페이지 정상 렌더 확인(비교 기준점).

## Phase 1 — 공통 레이아웃 정리 & top 리팩토링 (핵심, 2~4일)
> top은 모든 페이지가 공유하므로 **가장 먼저, 가장 안전하게** 효과를 봅니다.

**1-1. 공통 레이아웃 도입 (Thymeleaf Layout Dialect 권장)**
- 현재: 페이지마다 `<head>`·`<body>` 중복 + fragment 수동 호출.
- 개선: `templates/layout/base.html` 하나에 공통 골격(head/top/footer/스크립트)을 두고,
  각 페이지는 `content`만 채우는 구조로 전환.
- 의존성 추가: `nz.net.ultraq.thymeleaf:thymeleaf-layout-dialect`.
- 효과: 페이지당 수십 줄 중복 제거, top/footer 한 곳만 고치면 전체 반영.

**1-2. top(헤더) 마크업 현대화** (`fragments/top.html`)
- `<table>` GNB → `<header><nav>` + Flexbox.
- `MM_showHideLayers`(인라인 visibility 토글) → CSS `:hover`/`:focus-within` 드롭다운 또는 작은 vanilla JS.
- 인라인 `onmouseover` 제거 → 이벤트는 JS 파일로 분리(`static/js/layout.js`).
- 이미지 로고/버튼 중 교체 가능한 것은 SVG/CSS로.
- 모바일: 햄버거 메뉴 토글 추가(고정폭 1180px → `max-width` + 반응형).

**1-3. CSS 정리 시작**
- 새 스타일은 `static/css/refactor.css`(또는 `app.css`) 한 파일에 모으고, 기존 `cipsystems.css`는 점진적으로 걷어냄.
- 인라인 스타일을 클래스로 추출(특히 반복되는 `border-bottom:1px solid #ddd` 류).

**산출물:** 새 `base` 레이아웃 + 리팩토링된 top fragment. 전 페이지에 자연 반영.

## Phase 2 — 게시판 2~3개 디자인 리팩토링 (3~5일)
> 대상 추천: **회사뉴스(news) · 건설IT뉴스(itnews) · 자료실(reference)** — 구조가 같아 한 번 만든 패턴을 재사용.

**2-1. (디자인만) 목록/상세 마크업 현대화 — DB 없이 먼저**
- `news.html`의 `<table>` 글 목록 → `<ul>`/`<div>` 카드 또는 시맨틱 `<table>`(스타일은 CSS로).
- 페이지네이션: `/board/news` + `/board/news1`로 쪼개진 **두 템플릿을 하나로 통합**(우선 마크업만, 클릭은 추후 동적화).
- 검색창(현재 주석) 디자인 복구.
- 상세 페이지(`news_detail*`)는 **공통 상세 템플릿 1개**로 디자인 통일(데이터 바인딩은 Phase 3).
- 깨진 ASP 잔재(`<%=intNowPage%>` 등) 전부 제거.

**2-2. (선택, 권장) 정적 → 동적 데이터 구조로 전환**
> "디자인만"이 목적이면 건너뛰어도 되지만, 게시판은 결국 동적화가 필요하므로 **최소 골격만** 잡아두길 권장.
- `news_detail1~19`처럼 글마다 템플릿을 두는 구조를 **단일 `news_detail.html` + `?seq=` 파라미터**로 변경.
- 1차로는 DB 없이 자바 `List<Map>`/하드코딩 모델이라도 `th:each`로 루프 렌더 → 템플릿 28개 → 2~3개로 축소.
- 컨트롤러도 `getViewNewsDetail` 1개로 통합(`@GetMapping("/board/news_detail")` + `@RequestParam Long seq`).

**산출물:** 게시판 3종의 현대화된 목록·상세 디자인 + (선택) 템플릿 수 대폭 감소.

## Phase 3 — 게시판 백엔드 동적화 (향후 본 개발, 별도 문서로)
> Phase 1~2가 "보이는 것"이라면 여기는 "동작하는 것". **현 단계에선 문서화만** 해두면 됩니다.
- 계층 신설: `domain`(엔티티) · `mapper`(MyBatis XML 또는 JPA) · `service` · `dto`.
- 레거시 `original-source/board/*.asp`(news_write/modify/del_action)에서 **컬럼·테이블 명세 역추출**.
  - 컬럼 한글명 통일이 필요하면 사내 **alias-dict** 규칙 적용(예: 공종명 → WBS_NAME / wbsName).
- 목록 페이징/검색/조회수/첨부파일(이미 `AWS S3`, `Tika` 의존성 보유) 설계.

## Phase 4 — 미래 아키텍처 전환 설계 문서 (React / Nuxt.js / PostgreSQL)
> 지금 코드를 바꾸는 게 아니라 **"어디로 갈지"를 적어두는 단계.** 별도 설계 문서로 분리 권장.

**4-1. 프런트 분리 (React 또는 Nuxt.js)**
- 현재 Thymeleaf SSR → SPA/SSR 프런트로 분리하려면 백엔드가 **REST/JSON API**를 제공해야 함.
  - 다행히 보안은 이미 JWT 기반 REST 체인(`/core/api/v*/rest/**`)이 준비돼 있음 → API 신설 시 재활용 가능.
- **선택 기준 정리(문서화 포인트):**
  - *React(CSR/Vite)*: 관리자/대시보드성 화면, SEO 비중 낮을 때.
  - *Nuxt.js(Vue, SSR)*: **공개 홈페이지처럼 SEO·OG 태그·초기 로딩이 중요할 때 적합** → 현재 사이트 성격엔 Nuxt 쪽이 자연스러움.
- 전환 전략: ① 백엔드를 API 서버로, ② 페이지 단위 **스트랭글러(Strangler) 패턴**으로 한 화면씩 신 프런트로 이관.

**4-2. DB 전환 (Oracle → PostgreSQL)**
- 영향: `ojdbc11`/`orai18n` → `org.postgresql:postgresql`, 방언/시퀀스/페이징(`ROWNUM`→`LIMIT/OFFSET`) 차이.
- MyBatis라면 Mapper XML의 Oracle 전용 함수(`NVL`, `SYSDATE`, `(+)` 조인 등) 치환 목록 작성 필요.
- 듀얼 데이터소스(`base`/`ctalk`) 중 홈페이지가 실제 쓰는 스키마 식별 → 마이그레이션 범위 한정.
- 권장: **신규 게시판부터 PostgreSQL**로 시작하고 레거시는 점진 이관.

**4-3. 목표 아키텍처(예시)**
```
[ Nuxt.js(SSR) ]  ──HTTP/JSON──>  [ Spring Boot REST API ]  ──>  [ PostgreSQL ]
   공개 홈페이지                       /core/api/v1/rest/**          (게시판/회원/첨부)
                                       JWT 인증 (기존 체인 재사용)
```
