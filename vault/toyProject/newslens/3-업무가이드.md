## 1. 공통 그라운드룰

### 브랜치 · 커밋 · PR
- `main` 보호 — 직접 push 금지, **PR로만** 머지.
- 브랜치 네이밍: `[1]` → `feat/collect-*`, `[2]` → `feat/api-*`, 공통 → `chore/setup-*` / `feat/common-*`.
- 커밋: `type: 한 줄 요약` (`feat`/`fix`/`refactor`/`chore`/`docs`). PR은 작게, 상대가 리뷰 후 squash 머지.

### 패키지 구조 (feature 기반 — 라인 분리에 맞춤)
```
com.newslens
├── common      공통 응답(ApiResponse)·예외(GlobalExceptionHandler)·유틸
├── config      SpringAI·Redis·OpenAPI(Swagger)·(SecurityConfig 자리[~])
├── article     엔티티·리포지토리·Feed/Article/Category 컨트롤러·서비스·DTO   [2]
```

## 2. 공통 기반 — Day 1~2 (둘이 함께) `[공통]`

여기서 **계약을 확정**해야 이후 병렬 작업이 안 부딪힌다.

### 2-1. DB 스키마 (Flyway `V1__init.sql`) — 최우선 계약
```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE articles (
  id           BIGSERIAL PRIMARY KEY,
  source       VARCHAR(30)  NOT NULL,          -- 'guardian'
  source_id    VARCHAR(200) NOT NULL,          -- Guardian 기사 id (중복 판별 키)
  url          TEXT         NOT NULL,
  title_en     TEXT         NOT NULL,          -- 원문 제목
  title_ko     TEXT,                           -- 번역 제목
  summary_ko   TEXT,                           -- 한국어 3줄 요약
  content_en   TEXT,                           -- 원문 본문(전문)
  category     VARCHAR(40),                    -- LLM 태깅 (예: AI/ML, 인프라)
  published_at TIMESTAMPTZ,
  created_at   TIMESTAMPTZ  NOT NULL DEFAULT now(),
  embedding    vector(768),                    -- 한국어 요약 임베딩
  CONSTRAINT uq_articles_source UNIQUE (source, source_id)
);
CREATE INDEX idx_articles_published ON articles (published_at DESC);
CREATE INDEX idx_articles_category  ON articles (category);
CREATE INDEX idx_articles_embedding ON articles USING hnsw (embedding vector_cosine_ops);

CREATE TABLE article_tags (
  article_id BIGINT      NOT NULL REFERENCES articles(id) ON DELETE CASCADE,
  tag        VARCHAR(40) NOT NULL,
  PRIMARY KEY (article_id, tag)
);

-- [~ 스트레치] JWT 자리 예약 — 8일 코어 미사용
CREATE TABLE users (
  id            BIGSERIAL PRIMARY KEY,
  email         VARCHAR(120) NOT NULL UNIQUE,
  password_hash VARCHAR(100) NOT NULL,
  created_at    TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```
> 벡터 차원 **768** = `text-embedding-004` 기본. 인덱스는 HNSW + 코사인.

### 2-2. 나머지 공통 셋업
- [ ] **Docker Compose**: `pgvector/pgvector:pg16` + `redis:7`. 포트·볼륨·헬스체크. 앱은 처음엔 각자 로컬 실행.
- [ ] **Spring AI 셋업**(채택 시): `spring-ai-starter-model-google-genai` 의존성 + 무료 키 설정 (`project-id`/`location` 절대 금지 — [병합 11-1](./2-병합계획)).
- [ ] **공통 코드**: `ApiResponse<T>`, `GlobalExceptionHandler`, 패키지 스켈레톤, OpenAPI(Swagger) 기본 설정.
- [ ] **이벤트 타입 정의**: `NewArticlesEvent`(아래 3-3 계약).

**Day 1~2 완료 정의(DoD)**: `docker compose up`으로 Postgres+Redis 기동 → Flyway 마이그레이션 통과 → 빈 Spring Boot 앱이 두 DB에 붙고 `/actuator/health` UP.

## 3. 연동 계약 (두 라인이 만나는 지점) `[공통]`

라인 분리의 핵심. 이 세 가지만 지키면 서로 안 기다려도 된다.

### 3-1. API 계약 (개발자 2가 제공, 데모 UI/외부가 소비)
| 메서드 · 경로 | 설명 | 응답 핵심 |
|---|---|---|
| `GET /api/feed?category=&page=0&size=20` | 최신 피드(페이징, 카테고리 필터) | `ArticleSummary` 페이지 (Redis 캐싱) |
| `GET /api/articles/{id}` | 기사 상세 | `ArticleDetail` |
| `GET /api/categories` | 카테고리 + 기사 수 | `[{category, count}]` (Redis 캐싱) |
| `POST /api/search` | 자연어 RAG 검색 (body `{query}`) | `{answer, sources: ArticleSummary[]}` |
| `GET /api/subscribe` | SSE 구독 | 이벤트 스트림 (3-3) |
| `[~] POST /api/auth/signup` · `/login` | 회원·로그인 | `{accessToken}` |

- `ArticleSummary`: `{id, titleKo, summaryKo, category, tags[], publishedAt, url}`
- `ArticleDetail`: `ArticleSummary` + `{titleEn, contentEn}`

### 3-2. LLM 처리 계약 (개발자 1 내부, 요청 수 최소화)
- **1콜**: 영문 기사 → 구조화 출력 `ArticleEnrichment{ titleKo, summaryKo, tags[], category }` (번역+요약+태깅 합침).
- **1콜**: `summaryKo` 임베딩 → `vector(768)`.
- 기사당 **총 2요청**. 전문 번역 안 함(원문 `content_en` 보관). 근거: [병합 11-4](./2-병합계획).

### 3-3. 이벤트 계약 (수집 → 실시간/캐시)
- `NewArticlesEvent(int count, List<Long> articleIds)`
- **발행** `[1]`: 배치가 새 기사 저장 완료 후 `ApplicationEventPublisher`로 발행.
- **소비** `[2]`: ① SSE로 구독자에게 `new-articles` 푸시 ② feed/categories **Redis 캐시 무효화**.

## 4. 라인별 상세 태스크 · 체크리스트

### 개발자 1 — 수집 · 처리 · 인프라 (쓰기) `[1]`
- [ ] **Guardian API 클라이언트** — Content Search API 호출(섹션=technology, AI 키워드), JSON 파싱. *(주의: Guardian은 REST/JSON이라 **ROME 불필요**. ROME은 RSS 피드를 추가할 때만.)*
- [ ] **스케줄러** — `@Scheduled`. 간격은 RPM(분당 5~15) 고려해 **30~60분 권장**(15분이면 throttle 강화).
- [ ] **중복 제거** — `(source, source_id)` 유니크로 upsert, 신규만 처리.
- [ ] **LLM 처리** — `ArticleEnrichment` 1콜 + 임베딩 1콜 (3-2 계약). Spring AI `ChatClient`/`EmbeddingModel` 또는 직접 클라이언트.
- [ ] **레이트리밋 큐** — 신규 다건을 한 번에 쏟지 않게 throttle/backoff (Resilience4j RateLimiter 또는 지연 큐). RPM 초과 방지.
- [ ] **저장 + 이벤트 발행** — 트랜잭션 저장 후 `NewArticlesEvent` 발행 (3-3).
- [ ] **Docker Compose 마무리** + **배포**(Railway/Render) + 시크릿 주입.

**DoD `[1]`**: 스케줄러 1회 실행 → 신규 기사가 `title_ko/summary_ko/tags/category/embedding`까지 채워져 저장 → `NewArticlesEvent` 발행 로그 확인.

### 개발자 2 — 조회 · 검색 · 실시간 (읽기) `[2]`
- [ ] **Feed/Article/Category API** — 3-1 계약대로 구현 (페이징·필터·상세).
- [ ] **Redis 캐싱** — feed 목록·categories 캐시(TTL) + `NewArticlesEvent` 수신 시 무효화.
- [ ] **RAG 검색 `POST /api/search`** — 질문 임베딩 → pgvector 코사인 Top-K → 컨텍스트로 답변 생성 → `{answer, sources}`. Spring AI `RetrievalAugmentationAdvisor` 또는 수동 배선.
- [ ] **SSE `GET /api/subscribe`** — `SseEmitter` 레지스트리 관리, 이벤트 수신 시 `new-articles` 푸시, 연결 종료 시 자동 제거.
- [ ] **Swagger** — 전 엔드포인트 문서화.
- [ ] `[~]` **JWT** — `users` 활용 로그인/토큰, SecurityConfig 필터, SSE 토큰 검증(쿼리파라미터/쿠키). [병합 7장](./2-병합계획).

**DoD `[2]`**: 피드·검색·구독이 데모 UI(또는 Swagger/curl)에서 동작 — 새 기사 수집 시 SSE 토스트 + 캐시 갱신 확인.

> ⚠️ **유일한 선행 의존**: `[2]`의 RAG는 임베딩된 기사가 있어야 검증 가능 → `[1]`의 LLM 처리(또는 시드 데이터 몇 건)가 선행. 그 외엔 3장 계약만으로 독립 진행.
