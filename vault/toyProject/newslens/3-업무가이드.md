# 업무 가이드 *(작성 예정)*

> **상태: 작성 예정** — [2. 병합 계획](./2-병합계획)이 확정된 뒤 이를 기준으로 작성한다. 아래는 담을 내용의 뼈대(목차)다.

## 담을 내용 (예정 목차)

1. **공통 그라운드룰**
   - Git 브랜치 전략 (예: `main` 보호 + feature 브랜치), 커밋 컨벤션, PR 규칙
   - 패키지 구조 / 공통 응답·예외 포맷 / 코드 컨벤션
2. **공통 기반 (Day 1~2, 함께)**
   - DB 스키마 확정 (`articles` 원문+요약+태그+`vector(768)`, `users` 자리 예약)
   - Docker Compose (Postgres+pgvector, Redis)
   - Gemini 클라이언트 래퍼(또는 Spring AI 셋업) — 공용
3. **라인별 상세 태스크 · 체크리스트**
   - 개발자 1 (수집·처리·인프라): 수집 배치 → 중복제거 → LLM 처리 → 저장/이벤트 → Docker/배포
   - 개발자 2 (조회·검색·실시간): Feed/Articles/Categories API → Redis 캐싱 → RAG 검색 → SSE → Swagger
   - 각 태스크별 완료 정의(DoD)와 의존 관계(예: RAG는 Gemini 래퍼 선행)
4. **Day별 진행표** (병합 계획 9장 기반 상세화)
5. **연동 지점 / 인터페이스 계약** — 두 라인이 만나는 API·DB·이벤트 스키마 합의
6. **스트레치 / 시즌2 진입 조건** — JWT, Redis Pub/Sub, AWS·k8s

## 작성 전 선행 결정

- 라인 배정 (누가 개발자 1/2)
- 제안 항목(Spring AI·ROME·Flyway 등) 채택 여부
- 배포처 / 테스트 범위
