# 실시간 채팅 + 인프라 풀코스 — 기획과 로드맵

> "개발자라면 꼭 다뤄봤어야 할 것"을 한 프로젝트에서 단계별로 밟아보기 위한 토이프로젝트 기획서다. WebSocket 실시간 채팅을 만든 뒤, 그걸 그대로 **Docker → AWS → Kubernetes** 사다리에 얹어 배포·운영까지 경험하는 게 목표다.
> 작성일: 2026-06-17 · 스택: Node.js + Socket.IO + Vue 3 (Spring 대안 가능)

## 왜 이 프로젝트인가

토이라도 "개발자 기본기를 한 바퀴 도는 것"을 목표로 하면, 보통 다섯 가지가 핵심 통과의례다.

- **인증/인가** (세션·JWT·OAuth)
- **DB 설계 + 트랜잭션/동시성**
- **캐싱** (Redis 등)
- **비동기 처리 / 메시지 큐**
- **실시간 통신** (WebSocket)

이 중 **실시간 채팅**은 WebSocket·동시성·상태관리·인증을 한 번에 밟는 데다 결과물이 눈에 보여 동기부여가 좋다. 여기에 평소 궁금했던 **Docker·AWS·Kubernetes**를 얹으면, 관심 기술 네 가지를 "한 프로젝트에서, 단계별로" 가져갈 수 있다.

## 등장 기술, 한 줄로 감 잡기

처음 접하는 인프라 용어들의 최소 정의다.

| 기술 | 한 줄 정의 | 왜 쓰나 |
|------|-----------|---------|
| **Docker** | 앱 + 실행환경을 '컨테이너'로 통째 포장 | "내 PC에선 됐는데…" 문제 해결, 어디서든 똑같이 실행 |
| **Kubernetes (k8s)** | 컨테이너 수십~수백 개를 자동 배포·확장·복구하는 관리자 | 컨테이너가 많아지면 손으로 관리 불가 → 자동 오케스트레이션 |
| **AWS** | 클라우드 인프라 백화점 (EC2·S3·RDS·SQS…) | 서버를 사지 않고 빌려 씀, 실무 표준 |
| **WebSocket** | 서버↔클라이언트 실시간 양방향 통신 통로 | 채팅·알림처럼 "서버가 먼저 말 거는" 기능 |

핵심은 **학습 사다리**다. 어떤 프로젝트든 결국 이 순서를 타게 된다.

```text
Docker(포장) → docker-compose(여러 개 묶기) → Kubernetes(자동 운영) → AWS(진짜 배포)
```

## 무엇을 배우게 되나 (앱 단)

채팅 앱 자체를 만들면서 자연스럽게 다음을 다룬다.

- **WebSocket 양방향 통신** — 메시지 송수신, 연결/끊김 이벤트 처리
- **인증/인가** — 소켓 핸드셰이크 단계에서 JWT 검증 (일반 HTTP 인증과 타이밍·방식이 다르다)
- **동시성/상태관리** — 누가 어느 방에 접속 중인지, 온라인 유저 목록을 메모리에서 관리
- **영속성** — 메시지를 DB에 저장하고, 재접속 시 히스토리를 불러오기
- **확장성 감각** — 단일 서버 인메모리 상태에서 시작해, Redis Pub/Sub로 다중 서버에 대응

### 일부러 마주칠 "함정"들

배움은 막히는 지점에서 나온다. 이 프로젝트가 의도적으로 부딪히게 만드는 지점들이다.

- 일반 HTTP 인증과 **소켓 인증의 차이** (헤더를 못 쓰는 경우, 핸드셰이크 타이밍 문제)
- 새로고침 시 **연결 끊김/재연결** 처리와 중복 메시지 방지
- 단일 서버 인메모리 상태가 **서버를 늘리는 순간 깨지는** 경험 → 이게 Redis를 도입하는 진짜 동기가 된다

## 일정 — 하루 1~2시간 기준

주당 약 8~10시간, 앱 개발은 익숙하지만 인프라는 처음이라는 전제로 짠 일정이다.

**총 예상: 약 11~13주 (≈ 3개월).** 인프라가 처음이면 AWS·k8s 구간에서 시간이 늘어나는 게 정상이다.

| 주차 | 마일스톤 | 핵심 학습 | 결과물 (이때 멈춰도 OK) |
|------|---------|----------|------------------------|
| **W1~2** | 채팅 MVP | WebSocket 양방향, 소켓 JWT 인증 | 1:1 채팅이 브라우저에서 실시간 동작 |
| **W3** | 채팅 디테일 + DB | 방(room), 온라인 목록, 입력중 표시, 영속화 | 새로고침해도 대화가 남는 다인 채팅방 |
| **W4~5** | **Docker** | 컨테이너 개념, Dockerfile, docker-compose | 앱+DB+Redis가 `docker compose up` 한 방에 |
| **W6~8** | **AWS 배포** | EC2/ECS, RDS, ElastiCache, 보안그룹, 도메인 | 인터넷에 떠서 친구가 접속 가능 ⭐ |
| **W9** | 확장성 준비 | Redis Pub/Sub 어댑터 | 서버 2대가 메시지를 공유 |
| **W10~13** | **Kubernetes** | Pod/Service/Deployment, 스케일아웃, 무중단 배포 | k8s 위에서 자동 운영·복구되는 채팅 |

### 어디서 끊어도 "완성"이 남는다

각 구간 끝에 그 자체로 이력이 되는 결과물이 남도록 설계했다.

- **W3 끝** → "실시간 채팅 앱을 만들었다" (앱 개발자 결과물)
- **W5 끝** → "컨테이너로 포장했다" (Docker 이력)
- **W8 끝** → "내가 만든 게 진짜 인터넷에 떠 있다" — ⭐ 포트폴리오로 가장 강력한 지점
- **W13 끝** → "k8s로 운영까지 했다" (인프라 엔지니어 결과물)

### 미리 알아둘 시간 함정

- **AWS (W6~8)**: 결제카드 등록·IAM 권한·보안그룹 막힘으로 첫 배포에서 하루이틀 헤매는 건 정상이다. **프리티어 + 비용 알림**을 꼭 걸고 시작한다.
- **Kubernetes (W10~13)**: Pod/Service/Ingress 개념이 한 번에 들어오지 않는다. 로컬(`minikube`/`kind`)에서 충분히 굴려본 뒤 AWS EKS로 넘어가는 걸 권한다.
- 위 일정은 **순항 기준 최소치**다. 막히면 해당 주가 1.5배로 늘 수 있다.

## 빠른 길

시간이 쪼들리면 **W8(AWS 배포)까지를 1차 목표(약 2개월)**로 잡고 한 번 완성·발표한 뒤, k8s는 시즌2로 분리하는 게 현실적이다. 동기부여 측면에서도 "일단 인터넷에 떠 있는 결과물"을 빨리 만드는 편이 낫다.
