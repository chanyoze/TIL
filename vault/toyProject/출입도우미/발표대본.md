# ③ Gradle (빌드 표준화)
- **[화면]** 카드 3개: toolchain / Wrapper 9.5.1 / Maven Central (+ build.gradle·gradle/wrapper)
- **[대본]** "다음은 자동화 파이프라인의 기초 공사인 **빌드 표준화와 Gradle 마이그레이션**입니다. 기존 원본은 이클립스 개발 환경에 종속되어 있었고, 로컬 PC 경로가 하드코딩된 `build.ps1` 스크립트로 빌드를 수행했습니다. 이로 인해 다른 개발자나 GitHub Actions 러너에서는 아예 컴파일 자체가 불가능했습니다. 이를 해결하고자 **Gradle Wrapper 9.5.1**을 프로젝트에 탑재했습니다. **JDK Toolchain** 설정을 도입하여 JDK 25 컴파일 버전을 지정해두면 로컬 환경에 맞춰 경로를 자동 탐색하도록 개선했습니다. 또한, 프로젝트 저장소에 직접 커밋해 두었던 JNativeHook 라이브러리 파일들을 제거하고, **Maven Central**에서 버전 좌표를 선언적으로 다운로드받게 강제했습니다. 이제 터미널이나 클라우드 러너에서 `./gradlew build` 한 줄만 실행하면, 어떤 OS에서든 동일하고 재현 가능한 빌드를 완수할 수 있는 완벽한 빌드 표준을 세웠습니다."
- **[포인트]** toolchain · Wrapper · Maven Central · `./gradlew build`

# ④ 테스트 (테스트 가능한 구조)
- **[화면]** ① CI는 GUI 없는 헤드리스 환경 → ② 이 앱은 그대로 테스트 어려움 → ③ 순수 로직만 함수로 추출해 검증 / 파일 역할(build.gradle·운영코드·*Test)
- **[대본]** "다음 단계는 지속적 통합(CI)의 검증 기반이 될 **단위 테스트**의 도입입니다. 이 애플리케이션은 화면이 핵심인 Swing GUI 앱이며, 마우스 전역 후킹과 OS 레벨의 클립보드에 의존합니다. 하지만 깃허브 액션이 구동되는 클라우드 러너는 모니터나 마우스가 없는 **화면 없는 헤드리스(Headless) 환경**입니다. 이 상태에서 화면 컴포넌트나 클립보드를 직접 테스트하면 즉각 `HeadlessException` 예외 에러가 납니다. 따라서 테스트 가능성을 확보하기 위해, GUI와 순수 로직을 철저히 분리했습니다. 설정 파일을 파싱하는 핵심 기능을 정적 순수 함수인 `parsePrograms`로 분리하여 파일이나 화면 없이도 테스트할 수 있는 구조로 변경했습니다. 그리고 **JUnit 5** 프레임워크를 도입해 공백 제거, 쉼표 구분 등 파싱 규칙을 자동 검증했습니다. 핵심은 단순히 테스트 코드 수가 늘어난 것이 아니라, **화면이 없는 환경에서도 비즈니스 로직을 검증할 수 있도록 설계 자체를 유연하게 개선했다는 점**입니다."
- **[포인트]** 헤드리스 한계 · 순수 함수 분리 · JUnit 5

# 🎤 발표용 통 스크립트 (그대로 읽는 버전)

> 슬라이드를 넘기며 그대로 읽는 발표체. 〔 〕는 슬라이드 전환 신호.

CI/CD 토이 프로젝트 리뷰 시작하겠습니다. 먼저 해당 프로그램에 대해 간략하게 소개 드리면 제가 군생활을 방위로 성북노인종합복지관라는 곳에서 근무하였습니다. 당시 코로나 시기였던만큼 어르신들의 건강을 위해 모든 출입명부를 관리했어야 했습니다. 방문하신 적 있으신 어르신들은 개개인마다 출입카드가 있어 이를 찍으면 성함, 전화번호 등은 기록되지만 체온, 방문 사유 등도 추가로 기록을 해야했습니다. 출입이 몰리는 시간대가 되면 모든 출입 기록을 기록하는데 시간이 많이 소요되서, 어르신들도, 타자가 느린 직원분들도 어려움을 호소하셔서 이에 도움이 되고자 이맘때 혼자 만들었던 자바 스윙 앱입니다. 기능은 복잡할게 없고 자주 사용되는 방문 사유를 버튼 형식으로 미리 등록해두고 클릭해서, 빠르게 기록을 할 수 있도록 한게 주요 기능입니다. 당시에 아쉬움이 남았던 부분은 준비 기간이 짧아서 퀄리티가 떨어졌고 숙원 기능이던 버튼 누르면 클립 보드에 이식까진 됐지만 좌클릭 시 붙여넣기 하는 기능을 구현하지 못했었습니다. 이번 기회에 리팩토링 겸 숙원 기능 이식 거기에 CI/CD 인프라를 직접 붙여서 빌드·테스트·배포를 자동화하는데 사용했습니다.

〔목차〕
순서는 이렇습니다. 개요로 시작해서 리팩토링, 그리고 Gradle·테스트·CI·CD·배포까지 파이프라인을 차례로 쌓고요, 중간에 겪은 트러블슈팅과 정리로 마무리하겠습니다.

〔개요 — 시작점〕
먼저 출발점입니다. 원본은 문제가 꽤 많았습니다. 앞서 말씀드린 것처럼 기능은 단순 복사만 가능했었고, 빌드는 이클립스와 PowerShell 스크립트에 묶여 있었고, JDK 경로까지 코드에 박혀 있었습니다. 게다가 실행 환경에도 묶여 있어서, 자바(JDK)가 설치된 PC가 아니면 exe 자체가 실행되지 않았습니다 — 이 부분은 뒤에서 jpackage로 자바 런타임을 통째로 묶어 해결합니다. 이 상태로는 깃허브 러너가 빌드조차 못 하기 때문에, 오른쪽처럼 하나씩 대처하면서 자동화의 토대부터 다시 세웠습니다.

〔개요 — 아키텍처〕
전체 그림은 이렇습니다. 우선 코드의 리팩토링부터 시작했습니다. 작업이 완료된 코드는 Github에 커밋했고, Github 저장소에 Gradle 빌드, JUnit 테스트, 깃허브 액션 CI, jpackage를 통한 배포, 그리고 Github Pages에 다운로드 페이지까지 한 줄기로 이어집니다. 처음부터 서버를 안 쓰겠다는 계획은 아니었는데, 다운로드 페이지가 백엔드 없는 정적 페이지라 별도 서버를 구축하지 않고도 GitHub Pages로 띄울 수 있었고, 빌드·배포도 Actions가 이벤트마다 처리해 줘서 — 결과적으로 서버 없이 비용도 관리도 없이 자동화할 수 있었습니다.

〔리팩토링 — 원본 코드〕
리팩토링 과정은 크게 '코드 정리 및 아키텍처 개선'과 '오류 조치' 두 방향으로 진행했습니다. 먼저 원본 코드 상태에서의 '코드 정리와 아키텍처' 관점입니다. 기존 원본 코드는 단일 Main.java 파일에 UI 생성, 이벤트 처리, 외부 프로그램 연동까지 모든 역할이 몰려 있는 전형적인 God 클래스 구조였습니다. main 메서드 하나가 230줄에 달하는 스파게티 코드였죠. 특히 버튼 13개의 복사 이벤트 처리는 Java 람다식의 루프 변수 캡처 제약을 극복하지 못해, 완전히 동일한 세 줄짜리 코드를 13번이나 복사해서 이어 붙여 DRY(Don't Repeat Yourself) 규칙을 심각하게 위반하고 있었습니다. 여기에 프로젝트 루트에 껍데기뿐인 Button.java 더미 파일이나 컴파일조차 안 되는 Removed.java 같은 Dead Code(폐기 코드)들이 그대로 커밋에 방치되어 있었습니다. 저는 아키텍처 규칙과 소프트웨어 공학 표준을 준수하기 위해, 패키지를 나누고 관심사별로 책임을 분할하여 이러한 중복 및 폐기 코드를 전면 정리하는 작업부터 진행했습니다.

〔리팩토링 — 책임 분리〕
두 번째는 런타임 안정성과 사용자 경험을 보장하기 위한 '오류 조치' 관점입니다. 원본 코드에는 실제 환경에서 먹통을 유발하거나 실행을 방해하는 치명적인 에러들이 곳곳에 숨겨져 있었습니다. 가장 먼저 외부 출석 프로그램(ATTEND_RF.exe)을 강제로 실행하기 위해 사용한 p.waitFor() 메서드였습니다. 이 동기식 대기 코드로 인해 외부 프로그램이 완전히 닫힐 때까지 Java 스레드가 블로킹되어 정작 우리 앱 창이 뜨지 않고 시스템이 완전히 멈춰버리는 블로킹 버그가 있었습니다. 또한, 해당 외부 프로그램의 파일 경로를 코드 내부에 하드코딩해 두었기에, 다른 PC에 가서는 IOException 크래시를 유발하는 환경 종속성 문제가 있었습니다. 마지막으로 Swing GUI의 권장 규칙인 EDT(Event Dispatch Thread) 동시성 규약을 지키지 않고 메인 스레드에서 직접 UI 컴포넌트들을 호출해 화면 깨짐이나 데드락 위험에 노출되어 있었습니다.
이에 대한 조치로, 블로킹을 유발하던 외부 프로그램 강제 연동을 제거하고, 경로와 항목명 등은 config.properties 설정 파일로 완전히 외부화했습니다. 그리고 UI 컴포넌트 생성은 SwingUtilities.invokeLater()를 활용해 안전한 EDT 전용 비동기 스레드 영역으로 감싸 실행함으로써 런타임에 발생할 수 있는 교착 상태와 예외 에러들을 완전히 해결했습니다.

〔리팩토링 — 숙원 기능〕
이번에 제일 하고 싶었던 게 자동 붙여넣기입니다. 자바는 보안상 창 밖 클릭을 직접 못 잡고, 강제 후킹은 백신이 키로거로 오인합니다. 이게 제가 당시 구현을 못했던 문제였고 이번에 조치한 방식은 버튼을 누르면 잠깐 기다렸다가, 사용자가 입력칸을 클릭하면 Robot이 OS 레벨로 Ctrl+V를 쏘는 '지연 투척' 방식으로 풀었습니다. 후킹이 실패하면 자동으로 끄고 수동 붙여넣기로 폴백합니다.

〔Gradle〕
다음은 자동화 파이프라인의 기초 공사인 빌드 표준화와 Gradle 마이그레이션입니다. 기존 원본은 이클립스 개발 환경에 종속되어 있었고, 로컬 PC 경로가 하드코딩된 build.ps1 스크립트로 빌드를 수행했습니다. 이로 인해 다른 개발자나 GitHub Actions 러너에서는 아예 컴파일 자체가 불가능했습니다. 이를 해결하고자 Gradle Wrapper 9.5.1을 프로젝트에 탑재했습니다. JDK Toolchain 설정을 도입하여 JDK 25 컴파일 버전을 지정해두면 로컬 환경에 맞춰 경로를 자동 탐색하도록 개선했습니다. 또한, 프로젝트 저장소에 직접 커밋해 두었던 JNativeHook 라이브러리 파일들을 제거하고, Maven Central에서 버전 좌표를 선언적으로 다운로드받게 강제했습니다. 이제 터미널이나 클라우드 러너에서 ./gradlew build 한 줄만 실행하면, 어떤 OS에서든 동일하고 재현 가능한 빌드를 완수할 수 있는 완벽한 빌드 표준을 세웠습니다.

〔테스트〕
다음 단계는 지속적 통합(CI)의 검증 기반이 될 단위 테스트의 도입입니다. 이 애플리케이션은 화면이 핵심인 Swing GUI 앱이며, 마우스 전역 후킹과 OS 레벨의 클립보드에 의존합니다. 하지만 깃허브 액션이 구동되는 클라우드 러너는 모니터나 마우스가 없는 화면 없는 헤드리스(Headless) 환경입니다. 이 상태에서 화면 컴포넌트나 클립보드를 직접 테스트하면 즉각 HeadlessException 예외 에러가 납니다. 따라서 테스트 가능성을 확보하기 위해, GUI와 순수 로직을 철저히 분리했습니다. 설정 파일을 파싱하는 핵심 기능을 정적 순수 함수인 parsePrograms로 분리하여 파일이나 화면 없이도 테스트할 수 있는 구조로 변경했습니다. 그리고 JUnit 5 프레임워크를 도입해 공백 제거, 쉼표 구분 등 파싱 규칙을 자동 검증했습니다. 핵심은 단순히 테스트 코드 수가 늘어난 것이 아니라, 화면이 없는 환경에서도 비즈니스 로직을 검증할 수 있도록 설계 자체를 유연하게 개선했다는 점입니다.

〔CI〕
이렇게 구축된 빌드와 테스트를 기반으로 지속적 통합(CI) 파이프라인을 GitHub Actions를 통해 연동했습니다. 매번 개발자가 코드를 수정할 때마다 수동으로 테스트 명령어를 치는 귀찮음과 누락 실수를 방지하기 위해, 코드가 원격 저장소에 push되거나 Pull Request(PR)가 열릴 때마다 깃허브 Actions 러너가 깨어나 자동으로 ./gradlew build를 실행합니다. ubuntu 리눅스 러너가 컴파일을 수행하고 JUnit 단위 테스트를 실행해 코드의 결함 여부를 검증합니다. 검증 결과는 PR 상세 화면에 초록색 통과 표시나 README 배지에 즉각 실시간으로 반영됩니다. 덕분에 저희 기본 브랜치는 항상 품질과 동작성이 검증된 배포 가능한 상태를 보장받게 되었습니다.

〔CD〕
다음은 지속적 배포(CD) 파이프라인입니다. CD를 설계하면서 핵심은 '언제 배포할지(트리거)'와 '어떤 포맷으로 배포할지(산출물)'에 대한 기술적 의사결정이었습니다. 첫째, 배포 시점은 매 커밋마다 70MB가 넘는 무거운 리소스를 빌드하여 클라우드 비용을 유발하는 낭비를 막고 릴리스 히스토리를 깔끔하게 관리하기 위해 '버전 태그(tags: v*) push'를 트리거로 확정했습니다. 둘째, 배포 형태는 사용자 PC에 자바가 깔려있지 않아도 즉시 실행될 수 있는 'JRE 내장형 포터블 zip 압축파일'을 채택했습니다. jpackage의 exe 단독 파일은 껍데기에 불과하고 runtime 폴더가 필수적이며, MSI 설치 마법사는 WiX Toolset 설치나 관리자 권한 요구 등 배포 허들이 높아 포터블 zip 형태가 가장 유리했습니다. 태그가 발행되면 windows-latest 러너가 기동하여 jpackageImage 태스크를 수행하고, 생성된 산출물을 압축해 GitHub Releases에 버전 명세와 함께 자동으로 업로드해 줍니다.

〔배포 — 다운로드 페이지〕
다음은 사용자와 만나는 마지막 창구인 배포 및 다운로드 페이지입니다. 배포 파일을 올려둔 깃허브 Releases 화면은 일반 사용자가 접근해서 받기에 다소 복잡하고 불친절합니다. 이를 개선하기 위해 GitHub Pages를 연동하여 깔끔한 랜딩 웹페이지를 호스팅했습니다. 이때 자동화의 완성도를 높인 기전이 바로 '고정 주소 URL(latest download 링크)' 방식입니다. 빌드 zip 파일 이름에 구체적인 버전을 제외하고 EntranceHelper.zip으로 통일해 올림으로써, 새 버전이 출시되어도 다운로드 주소는 항상 latest/download/EntranceHelper.zip으로 동일하게 유지됩니다. 이 덕분에 개발자가 새 버전을 내더라도 다운로드 페이지 코드를 수정할 필요가 없으며, 태그 푸시라는 단 한 번의 액션으로 빌드, 검증, 패키징, 릴리스 업로드, 다운로드 페이지 링크 갱신까지 무인으로 한 번에 처리되는 무결한 완전 자동화 사이클을 갖추게 되었습니다.

〔트러블슈팅 — CI/CD〕
이번 프로젝트의 핵심 성과 중 하나인 CI/CD 트러블슈팅입니다. 빌드 인프라를 클라우드로 올리면서 크게 세 가지 예기치 못한 에러를 겪었습니다. 첫 번째는 릴리스에 460바이트 크기의 빈 zip이 올라간 조용한 실패 버그였습니다. 워크플로는 초록색 성공으로 떴지만 파일이 비어있었는데, 압축 단계에서 한글이 포함된 텍스트 파일을 읽어 경로를 재조립하는 셸 스크립트 과정 중 인코딩이 깨진 게 원인이었습니다. 게다가 Compress-Archive 커맨드는 대상이 없어도 예외 에러 없이 조용히 빈 zip을 생성해 탐지가 늦어졌습니다. 이를 조치하기 위해 실제 폴더를 직접 탐색하도록 바꿨고, 압축 명령어를 tar로 교체했으며, 1MB 이하의 산출물은 빌드를 강제 실패시키는 크기 가드를 심고 파일명을 영어로 바꿨습니다. 두 번째는 대문자 태그 미트리거 현상이었습니다. 태그를 대문자 V1.2.0으로 배포했으나 CD가 작동하지 않았는데, 트리거 규칙의 소문자 v*가 대소문자를 엄격히 구분하고 있었기 때문입니다. 트리거를 대소문자 모두 포함하도록 배열로 지정하고 기존 태그를 강제 재발행해 조치했습니다. 세 번째는 리눅스 CI 러너에서의 gradlew 실행 권한 오류였습니다. 윈도우에서 작업을 커밋하다 보니 실행 비트가 누락된 채 git에 올라가 생긴 문제였습니다. git update-index --chmod=+x gradlew 명령어로 깃 내부에 명시적 실행 권한을 부여해 해결했습니다. 이 과정들을 통해 성공 로그가 정상 동작을 보장하지 않으며, CI/CD 환경은 OS와 인코딩 차이에 극도로 민감하다는 교훈을 얻었습니다.

〔트러블슈팅 — 앱〕
다음은 앱 런타임 디버깅 및 버그 조치 과정입니다. 앱 고도화와 기능 확장 중에 두 가지 고질적인 버그를 해결했습니다. 첫 번째는 단축키가 최초 1회만 동작하고 먹통이 되는 포커스 버그였습니다. 1부터 9까지의 단축키가 한 번은 눌리는데, 미리보기 창을 마우스로 클릭하고 나면 입력이 막혔습니다. 분석 결과, 단축키 입력 이벤트 시 텍스트 창에 포커스가 가 있으면 단축키 대신 텍스트로 인식되어 단축키 처리가 양보되게 설계되어 있었는데, 양보 대상에 편집이 안 되는 미리보기 영역까지 포함되어 버그가 발생한 것이었습니다. 단축키 양보 대상을 오직 '사용자 정의 입력 칸'으로만 좁게 한정함으로써 포커스가 어디로 가도 단축키가 즉각 감지되도록 해결했습니다. 두 번째는 편집 아이콘이 네모 모양의 소위 '두부 현상'으로 깨져서 나오는 폰트 글리프 오류였습니다. 윈도우 OS의 표준 폰트인 맑은 고딕에 해당 연필 모양 ✎ 특수 문자의 글리프가 내장되어 있지 않아 생긴 문제였습니다. 이를 단일 글자 처리가 아닌, 실행 환경의 폰트가 해당 글리프를 표시할 수 있는지 canDisplay 메서드로 사전에 검증하여 표시 가능한 글자만 유연하게 부착하는 글리프 가드를 구현해 일반화했습니다. 특정 환경에 종속되지 않고 언제나 안전한 깨짐 방지 뷰를 확보할 수 있었습니다.

〔정리 — Before → After (앱 화면)〕
다음으로 리팩토링 전후의 실제 화면 비교입니다. 왼쪽이 기존의 단일 파일 기반으로 만들어진 원본 화면이고, 오른쪽이 현재의 v1.3 화면입니다. 단순히 화면이 현대적이고 깔끔해진 것을 넘어, 레이아웃 매니저를 도입하여 컴포넌트 겹침 문제를 해결했고, 하단 상태 표시줄로 후킹 상태와 마지막 복사 값을 한눈에 볼 수 있게 했습니다. 또한, 항목 편집 다이얼로그를 통해 언제든 버튼 목록을 커스터마이징하고, 정보 창을 통해 GitHub Releases API로 최신 버전 업데이트를 백그라운드에서 확인하는 현대적인 애플리케이션의 구색을 갖추게 되었습니다.

〔정리 — 배운 점〕
정리하겠습니다. 자동화로 반복과 실수가 줄었고, 테스트는 결국 구조의 문제라는 걸 배웠습니다. 단위 테스트와 CI, 산출물 가드로 회귀를 막았고, 작은 단위 PR과 문서화로 추적성을 확보했습니다. 결정이 구현보다 먼저였고, 토이 규모엔 무료 인프라로 충분했습니다. 환경 차이에 민감하다는 점도 크게 느꼈습니다.

〔마무리〕
이번 토이 프로젝트의 성과를 발판 삼아, 다음 단계로는 Docker 컨테이너 기술을 학습하고 파이프라인에 이식할 계획입니다. 비록 이 프로젝트는 데스크톱 바이너리 릴리스라 컨테이너 배포가 어울리지 않지만, 별도의 백엔드 컨테이너 저장소를 파서 GitHub Actions로 Docker 빌드를 수행하고, GitHub Container Registry인 GHCR에 이미지를 자동으로 push하는 웹 앱 CD 프로젝트를 구축해 볼 예정입니다. 이상으로 CI/CD 토이 프로젝트 리뷰를 모두 마치겠습니다. 들어주셔서 감사합니다.
