# 검증(차단) 메시지 우선순위 설계 원칙

> "삭제/수정/저장을 막을 때 **어떤 사유 메시지를 보여줄지**"를 일관되게 정하는 원칙.
> 여러 화면에 같은 검증이 흩어질 때 문구·순서가 제각각이 되는 문제를 막는다.

## 문제 상황

한 동작(예: 삭제)을 막는 사유가 여러 개일 수 있다.

- 권한이 없음
- 시스템이 만든(보호) 항목임
- 현재 상태에서는 불가(승인 진행 중 등)
- 하위 버전/연결 데이터가 있어 불가

`if ~ else if ~` 가드 체인으로 검사하면 **위에서 먼저 걸린 사유의 메시지만** 노출된다.
→ 화면마다 검사 순서가 다르면 **같은 상황에 다른 메시지**가 떠 사용자가 혼란스럽다.

## 핵심 원칙

> **차단 여부(가능/불가) 결과는 순서와 무관하다.** 순서는 오직 "어떤 메시지가 뜨는지"만 결정한다.

그러니 순서를 **"막는 사유의 근본성"** 기준으로 통일한다:

```
1) 권한 / 소유   ─ 가장 근본적 (애초에 건드릴 자격이 있나)
2) 항목 속성     ─ 시스템 보호 항목인가
3) 워크플로 상태 ─ 지금 단계에서 허용되는 동작인가
4) 종속 데이터   ─ 하위 버전/연결이 남아있나
```

## 적용 예시 (삭제 표준 순서)

```text
1. 삭제 권한 (보안등급 / 현장옵션)
2. 소유권   (다른 조직이 공유한 항목인가)
3. 수정 권한
4. 시스템 보호 항목 여부
5. 현재 상태 (승인대기/진행/완료면 불가)
6. 종속 버전 존재 여부
```

- 화면마다 해당 없는 단계는 **건너뛰되 상대 순서는 유지**한다.
  - 예: 일부 화면엔 "소유권/수정권한" 개념이 없으면 `1 → 4 → 5 → 6`.

## 부가 가이드

- **같은 의미의 컬럼이 화면마다 출처가 다를 수 있다.** (예: 같은 `삭제가능여부` 플래그가 한 화면은 "보안등급", 다른 화면은 "현장옵션" 기반)
  → 이때는 메시지 문구도 출처에 맞게 **의도적으로** 다르게 둔다("권한이 없는 보안등급" vs "권한이 없는 현장").
- **메시지 문구 자체도 표준화**한다: `"{상태명} 상태의 {대상}은 ~할 수 없습니다."` 같은 템플릿으로 통일.
- "메시지 없이 UI만 차단(버튼 hide / readOnly)"하는 경우와 "메시지로 안내"하는 경우의 기준도 정해둔다.

> 한 줄 요약: **막을지 말지는 로직, 무엇을 보여줄지는 순서.** 순서를 "근본성"으로 통일하면 전 화면 메시지가 일관된다.
