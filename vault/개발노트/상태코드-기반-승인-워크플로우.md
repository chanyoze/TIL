# 상태코드 기반 승인 워크플로우 설계 패턴

> 결재/승인이 여러 단계로 이어지는 화면을 만들 때 자주 쓰는 **상태머신 + 역할별 권한** 패턴.
> 실무 사례를 일반화한 노트 (내부 코드·도메인 무관).

## 1. 상태(Status)를 코드로 모델링

각 단계를 **숫자 코드**로 두고, "지금 어느 단계인가"를 한 컬럼(`STATUS`)으로 표현한다.

```
10  작성중        (draft)
20  승인대기      (submitted)
21  1차 승인대기  (예: 관리자)
22  2차 승인대기  (예: 팀장)
23  3차 승인대기  (예: 책임자)
24  외부 승인대기 (예: 발주처)
30  승인완료      (approved)
```

> 코드 구간을 의미 있게 쪼개면(예: `2x` = "승인대기 계열") **"승인대기 전체 조회"** 같은 필터를 `STATUS LIKE '2%'` 한 줄로 처리할 수 있다.

## 2. 단계 수는 "옵션 플래그"로 분기

같은 화면이라도 현장/회사마다 결재 단계 수가 다를 수 있다 → **옵션 플래그**로 흐름을 갈래낸다.

| 옵션 | 추가 시 초기 상태 | 흐름 |
|---|---|---|
| OFF | `20 승인대기` | 누구나 승인 → `30 완료` |
| ON | `21 1차 대기` | 1차 → 2차 → 3차 → (외부) → `30 완료` |
| ON + 외부승인 | … | 3차 승인 시 `24 외부대기` → 외부 승인 시 `30 완료` |

- "허가요청 단계를 건너뛰고 바로 등록"하는 화면은, 추가 시 초기 상태를 `작성중`이 아니라 **옵션에 따라 `1차/승인대기`로** 시작시킨다.

## 3. 권한은 "상태 × 역할"의 교차로 결정

수정/삭제/승인 가능 여부를 **현재 상태 + 사용자 역할**의 조합으로 판단한다.

```text
if STATUS == 21(1차 대기):  지정된 1차 승인자만 수정/삭제/승인
if STATUS == 22(2차 대기):  지정된 2차 승인자만 ...
if STATUS == 20(승인대기, 옵션 OFF): 누구나 가능
```

- "지정된 역할"은 행(row)에 저장된 담당자와 로그인 사용자를 비교.
- **승인 취소**도 같은 규칙(그 단계 담당자만)을 적용해 일관성 유지.

## 4. 화면 요소 visibility도 옵션과 연동

- 외부 승인(발주처 등) 옵션이 `OFF`면 관련 컬럼/버튼은 **숨김(visible=false)**, `ON`이면 노출.
- "분기 = 데이터(옵션)"로 두면 화면 코드를 건드리지 않고 동작을 바꿀 수 있다.

## 정리: 이 패턴의 장점

1. 상태가 **한 컬럼**에 모여 흐름 추적이 쉽다.
2. 단계 수/외부승인 유무를 **옵션 데이터**로 바꿔 다현장 대응.
3. 권한 로직이 `상태 × 역할` 표로 정리돼 테스트 케이스를 만들기 좋다.

> ⚠️ 주의: 상태 코드값과 의미를 **한 곳(코드 테이블/Enum)** 에 정의해두지 않으면 매직넘버가 코드 전체에 흩어진다.
